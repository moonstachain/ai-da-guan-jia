# OSA 完美任务方法论 v1

OSA 是 `AI大管家` 现有 `合作宪章` 的操作层翻译，不是平行新体系。

它把每轮任务固定为一条闭环：

1. 先过六个治理判断
2. 再写三层目标
3. 再做策略判断
4. 再做行动与管控编排
5. 最后过 closure gate，把结果和进化分开写清

## 适用边界

- 适用于需要 `AI大管家` 统筹的任务
- 适用于单任务、战略线程任务、高自治任务、高失真风险任务
- 不要求本轮先改脚本实现
- 不替代具体 skill，只约束具体 skill 进入任务前后的标准动作

## 一、起手式：六个治理判断

每轮开始前必须显式输出：

- `自治判断`
- `全局最优判断`
- `能力复用判断`
- `验真判断`
- `进化判断`
- `当前最大失真`

没有这 6 条，不算进入 OSA。

## 二、O / Objective

`O` 负责把任务放回同一条目标链里：

- `原目标对齐`
  - 说明任务如何服务 `合作宪章` 与 `meta-constitution`
  - 说明任务除了当下结果，还会为递归进化增加什么
- `战略目标对齐`
  - 说明该任务对应哪个阶段目标或治理线程
  - 说明不做这件事会拖慢哪条战略主线
- `任务目标定义`
  - 交付物是什么
  - 什么算完成
  - 什么明确不做
- `失真边界`
  - 这轮最可能出现的伪完成是什么
  - 用什么规则阻止它

## 三、S / Strategy

`S` 负责在执行前先把路径和证据说清：

- `外部对标`
  - 需要参考哪些外部资料、案例、最佳实践
- `既往经验萃取`
  - 复用哪些已有 skill、脚本、auth、历史工件
- `可行性评估`
  - 成本、依赖、风险、验证难度
- `路径决策`
  - 为什么选当前路径，而不是更熟悉或更花哨的路径
- `验真目标`
  - 最终必须交付哪些证据才能证明不是假完成

## 四、A / Action

`A` 负责把执行变成最小充分编排，而不是工具堆叠：

- `任务拆解`
  - 子任务顺序和依赖
- `能力编排`
  - 每个子任务使用什么 skill、脚本、workflow 或人工
- `管控节点`
  - 哪些节点必须停下来做验真、校准或升级
- `人机边界`
  - 仅在登录、授权、付款、不可逆发布或删除、不可替代主观判断时打断共同治理者
- `闭环输出`
  - 最终结果
  - 证据
  - 复盘
  - 下一轮迭代候选

### A 阶段选型规则

默认选型顺序固定为：

1. `已有 skill`
2. `本地脚本/CLI`
3. `已登录的外部 workflow`
4. `人工介入`

排序规则固定沿用现有路由政策：

`任务适配度 > 验真能力 > 成本/算力 > 登录态复用 > 新增复杂度`

默认 downstream skill ceiling 为 `<= 3`。
超过 3 个必须显式说明为什么省不掉。

## 五、Closure Gate

OSA 名称不变，但每轮执行结束后必须追加强制收尾门：

- `结果是否达成`
- `证据是什么`
- `gain`
- `waste`
- `next iterate`
- `rule candidate`

没有 closure gate，只能算动作结束，不能算真正完成。

## 六、通过标准

任一进入 OSA 的任务都必须满足：

- 能写出 `原目标 / 战略目标 / 任务目标`
- 能写出最小执行链，而不是只报工具名
- 能写出明确验真证据，而不是口头完成
- 能写出 `gained / wasted / next iterate`

## 七、固定产物

OSA v1 在本仓库固定保留以下工件：

- 方法论主文档：`docs/osa-methodology-v1.md`
- 接口映射附录：`docs/osa-interface-mapping-v1.md`
- 机器可校验结构：`specs/osa/osa-card.schema.json`
- 真实样例：`specs/osa/examples/task-adagj-001-osa.json`

## 八、默认假设

- `OSA` 名称保持不变
- 复盘与递归进化通过 closure gate 进入同一闭环
- 先做完整操作规范，不先穷举工具矩阵
- 先把方法和接口映射定清，不先碰运行时脚本
