# 原力OS 协同治理仪表盘 Prompt Pack v1

## 1. 产品定义

- 产品外显名：`原力OS`
- 产品页面名：`原力OS 协同治理总控舱`
- 生成目标：给 `妙搭` 一套可以直接生成页面的完整提示词，不依赖它自己推断原力OS语义
- 页面形态：`单页总控舱`
- 目标受众：
  - 你本人
  - 后续和你一起运营原力OS的人类协作者
- 页面定位：
  - 这是一个人类运营驾驶舱
  - 不是普通 BI 报表
  - 不是康波时间轴换皮
  - 不是后台管理员配置台

## 2. 使用方式

推荐按下面顺序喂给 `妙搭`：

1. 先贴“主 Prompt”
2. 再贴“数据绑定附录”
3. 再贴“卡片映射附录”
4. 如果首轮结果跑偏，再贴对应“纠偏 Prompt”
5. 最后按“验收清单”复核

## 3. 主 Prompt

```text
你现在要设计并生成一个中文的高完成度单页驾驶舱，页面名称叫《原力OS 协同治理总控舱》。

先理解产品定义：
- 原力OS 是 AI大管家 的人类外显层。
- 它不是泛化 BI 系统，不是企业普通经营看板，也不是历史时间轴叙事页。
- 它是一个“人机协同治理驾驶舱”，用来帮助人类每天判断：现在系统怎样、为什么这样、下一步该做什么。

页面目标：
- 让人一页看懂当前原力OS治理运行态
- 让人立刻看到前台主线、阻塞边界、待我处理项和下一步动作
- 页面必须同时满足：高级、漂亮、克制、可运营、可读

风格要求：
- 借鉴“300年康波周期分析仪表盘”的高级感、层次感、中心重力和沉浸感
- 但必须治理化转译，不能照搬康波的宏大叙事或历史时间轴
- 整体采用深色分析舱风格，不用纯黑，背景偏 slate / ink
- 使用少量辉光、玻璃感和精细边框，但保持克制，不要做成炫技型大屏
- 强调色只允许三组：
  - cyan/blue：运行中、正常推进
  - amber：待我处理、需要人类输入
  - rose/red：系统阻塞、严重异常
- 文案必须短、准、结论先行，不要长段解释

页面结构固定为三层：

第 0 行：Hero 区
- 大标题：原力OS 协同治理总控舱
- 副标题要解释：原力OS 是 AI大管家 的人类外显层，用于协同治理与运营总控
- 展示最近刷新时间
- 展示当前运行态 badge
- 展示当前前台 focus badge

第 1 行：4 个 KPI 卡
- 当前前台主线数
- blocked_needs_user 数
- blocked_system 数
- 数据源联通健康度
- 这四张卡要像高级产品首页 summary，一眼看懂，不做复杂图表

第 2 行：总览与诊断层，固定 4 张卡
1. 前台主线卡
2. 任务状态分布卡
3. 数据源同步健康卡
4. 最近治理推进卡

第 3 行：行动层，固定 4 张卡
1. 待你处理
2. 系统阻塞队列
3. 下一步动作队列
4. 能力与审批摩擦卡

信息架构规则：
- 页面阅读顺序必须是：overview -> diagnosis -> action
- 一张卡只回答一个治理问题
- 每张卡都要有明确的标题、副标题、信息重点、视觉形态
- 任何诊断卡都必须能自然导向一个动作
- 页面中部要有“中心重力视觉”，但不要做历史时间轴；要改成治理分段流图、主线状态图或治理流态视图

非目标：
- 不要生成普通浅色企业报表
- 不要生成传统 BI 的密集表格拼盘
- 不要生成康波式长叙事故事页
- 不要假设系统已经自动完成复杂联动、卡片自动创建或精细化 Feishu widget 绑定

输出要求：
- 先给完整页面结构和视觉方案
- 再给每张卡的组件建议、数据字段建议和交互建议
- 如果你无法直接绑定数据，也要先生成页面壳、卡片结构和示意绑定点
- 所有设计说明必须用中文
```

### 4.4 建议派生指标

- `frontstage_thread_count`：前台主线视图中的对象数
- `blocked_needs_user_count`：`需要人类输入 != 空` 的对象或任务数
- `blocked_system_count`：`对象状态 = blocked_system` 或同义系统阻塞数
- `source_health_score`：
  - `missing_sources_snapshot = none`
  - `remote_reachable = true`
  - `github_backend = git_ssh`
  - 以上条件满足越多，健康度越高
- `recent_governance_velocity`：最近 review / decision / writeback 的发生频率
- `approval_friction_score`：能力项中 `requires_human_approval = true` 且高复用、高验真的聚合信号

## 5. 卡片映射附录

### Row 0：Hero

- 问题：这是什么系统、现在处在什么运行态、今天主会场在哪里
- 视觉建议：大标题 + 副标题 + 2 个 badge + 更新时间
- 主数据来源：
  - `总控对象主表`
  - `Review批次表`
  - `决策记录表`

### Row 1：KPI

#### KPI-1 当前前台主线数

- 治理问题：当前主会场有几条真正前台推进的主线
- 来源：`总控对象主表`
- 重点字段：`运营分段`、`对象状态`
- 建议形态：发光数字卡 + 小型趋势提示

#### KPI-2 blocked_needs_user 数

- 治理问题：现在有多少事项真的只差我输入
- 来源：`总控对象主表` 或 `任务总表`
- 重点字段：`需要人类输入`、`人类边界状态`
- 建议形态：amber 状态数字卡

#### KPI-3 blocked_system 数

- 治理问题：当前系统层阻塞有多少
- 来源：`总控对象主表`
- 重点字段：`对象状态`、`阻塞原因`
- 建议形态：rose 状态数字卡

#### KPI-4 数据源联通健康度

- 治理问题：整个协同治理的数据底座是否健康联通
- 来源：`数据源同步表`
- 重点字段：`remote_reachable`、`missing_sources_snapshot`、`github_backend`
- 建议形态：圆环或进度态健康卡

### Row 2：总览与诊断

#### 卡 1：前台主线卡

- 问题：当前真正推进中的主线是什么
- 来源：`总控对象主表`
- 推荐视图：`前台主线`
- 重点字段：
  - `对象标题`
  - `对象状态`
  - `当前摘要`
  - `下一步动作`
- 建议形态：高信息密度卡片列表，不要传统表格

#### 卡 2：任务状态分布卡

- 问题：任务整体分布如何，堵点集中在哪
- 来源：`任务总表`
- 重点字段：
  - `状态`
  - `runner_state`
  - `人类边界状态`
  - `优先级`
- 建议形态：横向条形图或分组堆叠图

#### 卡 3：数据源同步健康卡

- 问题：三端一网和同步 transport 是否稳定
- 来源：`数据源同步表`
- 重点字段：
  - `状态`
  - `最近扫描时间`
  - `remote_reachable`
  - `github_backend`
  - `missing_sources_snapshot`
- 建议形态：多状态矩阵 + 状态标签

#### 卡 4：最近治理推进卡

- 问题：系统最近做了哪些关键 review / decision / writeback
- 来源：
  - `Review批次表`
  - `决策记录表`
  - `治理动作与写回表`
- 重点字段：
  - `评审日期`
  - `摘要`
  - `decision_summary`
  - `event_time`
  - `verification_state`
- 建议形态：克制的纵向活动流，不是历史大时间轴

### Row 3：行动层

#### 卡 1：待你处理

- 问题：现在你必须介入什么
- 来源：`总控对象主表` 或 `任务总表`
- 过滤：`需要人类输入 != 空`
- 重点字段：
  - `对象标题`
  - `需要人类输入`
  - `阻塞原因`
  - `下一步动作`
- 建议形态：优先级队列卡

#### 卡 2：系统阻塞队列

- 问题：哪些问题不是人没做，而是系统本身在挡路
- 来源：`总控对象主表`
- 过滤：`对象状态 = blocked_system`
- 重点字段：
  - `对象标题`
  - `阻塞原因`
  - `证据入口`
- 建议形态：异常队列卡，强调红色状态点

#### 卡 3：下一步动作队列

- 问题：如果我现在开始推进，优先动作是什么
- 来源：`总控对象主表`
- 重点字段：
  - `对象标题`
  - `负责人模式`
  - `对象状态`
  - `下一步动作`
- 建议形态：短句行动清单

#### 卡 4：能力与审批摩擦卡

- 问题：哪些能力高价值但审批/授权摩擦大
- 来源：`技能与能力表`
- 重点字段：
  - `名称`
  - `verification_strength`
  - `auth_reuse`
  - `routing_credit`
  - `requires_human_approval`
- 建议形态：二维比较卡或气泡卡，突出高价值高摩擦项

## 6. 纠偏 Prompt

### 6.1 如果首轮结果“够漂亮但不够治理”

```text
请不要把这页当成品牌展示页或概念海报。它首先是一个每天要被使用的人类治理驾驶舱。

请强化以下几点：
- 每张卡都要回答一个明确治理问题
- 每张卡都要能映射到真实表和真实字段
- 页面要先回答：现在怎样、为什么、下一步做什么
- 弱化装饰性视觉，增强状态、分段、阻塞和行动可读性
- 不要出现无法落到真实数据的抽象文案卡
```

### 6.2 如果首轮结果“太企业 BI，缺少高级感”

```text
当前结果过于像传统企业 BI，看起来平、硬、散，缺少原力OS应有的高级完成度。

请往以下方向收敛：
- 保留深色分析舱底色
- 增加克制的玻璃感、光晕边框和空间层次
- 提高 Hero 与 KPI 的产品感和品牌完成度
- 强化页面中部的中心重力视觉
- 保持克制，不要走炫技大屏路线
- 目标是接近“300年康波仪表盘”的完成度，但更冷静、更治理化
```

### 6.3 如果首轮结果“没接住原力OS概念”

```text
你当前把这个页面理解成了普通任务管理看板，但它真正的概念层是：原力OS 是 AI大管家 的人类外显层，是一套把线程、任务、能力、来源、决策和写回组织在一起的人机协同治理系统。

请重新组织页面语义：
- Hero 必须明确说明原力OS与AI大管家的关系
- 页面重点不是“项目列表”，而是“治理运行态”
- 重点对象包括：主线、阻塞、来源健康、决策推进、行动队列
- 页面要像 operating system cockpit，而不是 project tracker
```

## 7. 验收清单

- 看到页面后，用户能在 10 秒内回答“现在系统总体怎样”
- 再看 20 秒，能回答“为什么这样”
- 再看 20 秒，能回答“下一步该做什么”
- 页面结构是固定单页，不是多页乱跳
- 页面包含：
  - Hero
  - 4 个 KPI
  - 4 张总览/诊断卡
  - 4 张行动卡
- 每张卡都能映射到 `协同治理` base 的真实表与真实字段
- 页面有高级感，但不是康波叙事页
- 页面可读、可运营，不是普通浅色企业报表
- 没把白皮书正文整段塞进 dashboard
- 没假装系统已经支持复杂自动联动、卡片自动创建或精细 Feishu widget 布局

## 8. 自动化边界说明

- 优先让 `妙搭` 使用 `协同治理` base 作为数据源
- 如果不能直接稳定绑定，就先生成页面壳、卡片结构和示意绑定点
- 这轮 prompt 只负责：
  - 页面概念
  - 信息架构
  - 视觉风格
  - 数据映射

  - Feishu dashboard 卡片自动创建

  - 白皮书语义自动理解
