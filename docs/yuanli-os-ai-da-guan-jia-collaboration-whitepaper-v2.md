# 原力OS × AI大管家 人机协同白皮书 v2

> 这是基于当前已实现版本的正式协同白皮书。  
> 如果你现在更需要一份“直接照着用”的日常手册，优先看 [docs/yuanli-os-xiaomao-ai-da-guan-jia-practical-guide-v1.md](/Users/liming/Documents/codex-ai-gua-jia-01/docs/yuanli-os-xiaomao-ai-da-guan-jia-practical-guide-v1.md)。  
> 它回答四个长期问题：
>
> - 你在飞书里的 `原力OS` 到底该怎么用
> - 哪些事应该交给后台 `AI大管家`
> - 哪些事必须留在人类边界
> - 什么叫“真闭环”，什么只是“聊得顺”

## 1. Summary

当前版的根定义固定为：

- `原力OS`
  - 你在飞书移动端里的连续陪伴前台
  - 负责高频、低风险、短链路的人机协同
- `AI大管家`
  - 唯一后端治理核
  - 唯一 canonical source of truth
  - 唯一闭环 owner
- `人类`
  - 提目标
  - 给偏好
  - 做授权
  - 做不可逆与主观取舍
  - 做最终验收

这份 v2 白皮书不是加新功能，而是把现有版本已经成立的能力，收敛成一套长期稳定的人机协同协议。

一句总纲写死为：

1. `你只需要自然表达目标。`
2. `原力OS 负责把高频协同变得顺手。`
3. `AI大管家 负责把事情做对、做真、做闭环。`

## 2. 六个判断

每次协同时，默认先做这六个判断：

- `自治判断`
  - 前台只承接高频、低风险、强反馈、短链路协同
  - 重执行、强验真、长链任务继续留在后端
- `全局最优判断`
  - 最优结构不是把全部能力塞进聊天窗
  - 而是 `飞书原力OS 前台 + AI大管家 后端 + 人类边界授权`
- `能力复用判断`
  - 复用当前 scenes、canonical capture、knowledge read path、close-loop 规则
  - 不重造第二套协作语言
- `验真判断`
  - 成功标准不是“更像人在聊天”
  - 而是你只靠手机就能发起、续接、追踪、轻判断、轻记录、准备收口
- `进化判断`
  - 先固化协同协议
  - 再迭代体验层与系统级入口
- `当前最大失真`
  - 最危险的误判是：`飞书里聊得顺 = 系统已经闭环`

## 3. 系统定位与单一根

### 3.1 单一根原则

- `AI大管家` 是唯一 canonical source of truth
- `原力OS` 不是第二大脑
- `原力OS` 不是第二闭环系统
- `原力OS` 不是第二任务库
- 飞书聊天记录不是 canonical memory，只是前台交互面

### 3.2 三层角色分工

| 层 | 角色 | 核心职责 | 不该做 |
| --- | --- | --- | --- |
| 人类 | 最终决策者 | 提目标、补充偏好、授权、审批、验收 | 不应该手工在大量技能与系统之间来回挑工具 |
| 原力OS | 移动前台 | 轻路由、轻结构化、轻摘要、轻判断、轻记录、恢复上下文、把你送到正确下一步 | 不抢长链执行、强验真、重集成、多系统复杂写入 |
| AI大管家 | 后端治理核 | route、skills 调度、知识查询、执行治理、证据归档、状态同步、close-loop 判断、run owner | 不把前台壳误当 source of truth，也不把聊天成功当完成 |

### 3.3 单一升级路径

- 前台能处理就前台处理
- 需要长链执行或强验真就自动转后端
- 遇到不可逆、人类偏好、权限或风险边界就明确升级给你

## 4. 飞书前台协同协议

### 4.1 原力OS 适合承接什么

当前版 `原力OS` 在飞书里承接这些高频动作：

| 场景 | 你可以怎么说 | 原力OS 应该做什么 | 典型返回 |
| --- | --- | --- | --- |
| 首页 / 唤醒 | `原力原力` | 告诉你现在能做什么、下一步怎么开口 | `scene + summary + next_step` |
| 恢复上下文 | `继续昨天那个` | 恢复最近 run、说明上次停在哪、下一步是什么 | `run_id + summary + next_step` |
| 我的任务 | `我现在有哪些任务` | 返回活跃任务、阻塞、待拍板、优先级 | `summary + next_step + verification_status` |
| 今日重点 | `今天最该看什么` | 给出当前最值得看的目标、阻塞或待拍板 | `summary + next_step + verification_status` |
| 任务入口 | `给我接个任务` 或直接描述任务 | 压成结构化推进链，生成 run | `run_id + summary + next_step + human_boundary` |
| 轻判断 | `帮我做判断 ...` | 做最小判断，不进入重执行 | `summary + next_step + human_boundary` |
| 轻知识 | `原力原力 搜一下 ...` / `查一下 ...` | 先选知识源，再给最小可证结论 | `summary + next_step + verification_status + run_id` |
| 轻记录 | `原力原力 记一下 ...` | 写入 canonical capture，返回 capture 摘要 | `summary + next_step + verification_status` |
| 收口准备 | `把这事闭环` | 按证据判断 ready / not_ready / blocked | `summary + next_step + verification_status + run_id` |
| 交给 PC | `这件事交给 PC 继续` | 告诉你为什么该切 PC、接力点在哪 | `summary + next_step + run_id` |
| 审批边界 | `这条消息要不要发` | 把人类边界说清楚，给建议但不替你拍板 | `summary + next_step + human_boundary + run_id` |

### 4.2 前台的职责边界

`原力OS` 负责这四件事：

1. 帮你把意图压成正确下一步
2. 帮你恢复最近 run、session、capture、任务状态
3. 帮你在手机端完成“看、问、定、记、催、转”
4. 帮你把复杂度挡在后台，不把系统内部语言甩给你

`原力OS` 不承诺这些事情：

- 深度研究
- 浏览器自动化
- 代码执行
- 批量写入
- 多系统强集成
- 不可逆动作审批

### 4.3 前台统一回复协议

当前版前台统一回复字段固定为：

| 字段 | 含义 |
| --- | --- |
| `scene` | 当前命中的前台场景 |
| `status` | 当前这一跳的处理状态 |
| `run_id` | 关联的任务 run 标识 |
| `session_id` | 当前前台会话标识 |
| `summary` | 人类可读的最小结论 |
| `next_step` | 现在最该做的下一步 |
| `human_boundary` | 是否已经触达人类边界 |
| `verification_status` | 当前证据状态 |
| `text` | 文本回复 |
| `card` | 结构化卡片 |

前台输出文风固定为：

- 所有用户可见标题统一为 `原力OS · ...`
- 输出只做 `短总结 + 明确下一步 + 边界提醒`
- 不在飞书里模拟完整后台推理
- 不把聊天窗变成长报告

### 4.4 前台默认使用规则

- 有唤醒词更顺手，无唤醒词也必须可用
- 一条消息允许多条前台意图 bundling
- `记一下` 必须产出 `capture_id` 并进入 canonical capture
- `搜一下` 优先回本地 canonical 与已接通读路径，不伪造外部已查结果
- `把这事闭环` 只能按证据返回 `ready / not_ready / blocked_*`
- 卡片只做结构化展示，不承载复杂事务流程

### 4.5 飞书里的建议说法

下面这些句子就是当前版的推荐用法：

- `原力原力 今天最该看什么`
- `继续昨天那个`
- `我现在有哪些任务`
- `原力原力 记一下：明天确认 Minutes 凭证`
- `原力原力 搜一下 Minutes 凭证`
- `帮我做判断：这件事先推进 A 还是 B`
- `把这事闭环`
- `这件事交给 PC 继续`

## 5. 后台治理与闭环协议

### 5.1 AI大管家 后端承接什么

后台 `AI大管家` 永远承接这 7 类工作：

1. `任务理解与 route`
2. `skills / 工具 / 知识源编排`
3. `长链执行`
4. `证据归档与 canonical 写入`
5. `状态同步与 run 管理`
6. `close-loop 判定`
7. `治理与演进`

### 5.2 后端永远负责的事实

- 把所有 substantive turn 绑定到 `run_id`
- 决定是否需要 `knowledge / execution / review / closure`
- 维护 canonical task、thread、capture、evidence
- 给出真实状态，而不是安慰性状态

### 5.3 后端输出状态

后端正式状态统一为：

| 状态 | 含义 |
| --- | --- |
| `completed` | 本地可完成部分已经闭环，证据充足 |
| `blocked_needs_user` | 只差你补一个明确输入、批准或选择 |
| `blocked_system` | 系统链路、权限、外部依赖或 runtime 被卡住 |
| `failed_partial` | 已完成一部分，但有明确失败残留，需要说明恢复点 |

### 5.4 后端与前台的固定分工

- 前台负责：`让你更快到正确下一步`
- 后端负责：`确保这个下一步真的对、可证、可追、可收`

### 5.5 什么时候自动切后台

出现下列情况时，默认从 `原力OS` 切到 `AI大管家`：

- 任务进入长链执行
- 需要深度研究
- 需要代码改动或文件工作
- 需要浏览器操作
- 需要多系统同步
- 需要强校验
- 需要正式 close-loop 证据判断

切换时的对外表达规则：

- 只用人类听得懂的话说明为什么要切
- 不暴露内部复杂路由细节
- 必须给你明确接力点

## 6. 人类边界与协同剧本

### 6.1 人类在原力OS 里最适合做的事

- 提目标：`我想把这件事推进到什么结果`
- 做选择：`A / B 哪个更符合我偏好`
- 补上下文：`这次优先 mobile first`
- 追状态：`现在卡哪了`
- 记灵感：`原力原力 记一下 ...`
- 做轻判断：`先看什么`
- 发起收口：`把这事闭环`

### 6.2 人类在 AI大管家 里最适合做的事

- 交付复杂任务
- 做强约束的项目推进
- 开启多轮规划、研究、写作、代码、批处理
- 看证据链、风险和治理结论
- 升级策略、标准和体系

### 6.3 必须升级给人类的边界

以下情况必须由人类拍板：

- 授权
- 发布
- 删除
- 付款
- 发外部消息
- 改生产配置
- 明显主观审美取舍
- 利益权衡或高风险商业判断
- 证据不足但有人想强行宣布完成

### 6.4 推荐的日常协同节奏

| 时段 | 最适合在原力OS 做什么 |
| --- | --- |
| 早上 | `今天最该看什么` |
| 白天碎片时间 | `原力原力 记一下`、`搜一下` |
| 推进中 | `我现在有哪些任务`、`继续昨天那个` |
| 临近收尾 | `把这事闭环` |
| 需要深工时 | `这件事交给 PC 继续` |

白皮书建议固定四个协同心智：

1. `我先在飞书发起`
2. `需要深工再切 PC`
3. `一切以证据为准`
4. `最终授权还是我来`

### 7.2 快速判断表

- 如果 `高频 + 低风险 + 手机端就能完成`，前台优先
- 如果 `长链 + 高复杂度 + 需要工具执行`，后端优先
- 如果 `高风险 + 不可逆 + 主观性强`，人类优先

## 8. 真闭环与证据规则

### 8.1 真闭环的定义

当前版白皮书把“真闭环”定义为同时满足：

1. 有目标
2. 有执行
3. 有证据
4. 有状态
5. 有边界说明

### 8.2 轻记录不是闭环

`原力原力 记一下 ...` 的定位必须被写死：

- 它是 capture
- 不是任务完成
- 可以被后续 `搜一下` 命中
- 可以为未来收口提供上下文
- 但本身不等于 evidence completion

### 8.3 收口的证据驱动规则

| 收口结果 | 含义 |
| --- | --- |
| `ready` | 已有足够 evidence，可以进入正式收口 |
| `not_ready` | 上下文存在，但证据还不够 |
| `blocked_needs_user` | 缺你提供的信息、批准或选择 |
| `blocked_system` | 系统链路、权限或外部依赖阻塞 |

### 8.4 白皮书里禁止出现的错误说法

- “聊完就算做完”
- “飞书消息就是记忆库”
- “原力OS 已经可以替代 AI大管家”
- “没有证据也能先算闭环”

## 9. Runtime 真值与信任边界

### 9.1 为什么要写 runtime truth

前台体验成立，不等于 live runtime 一定干净。  
因此白皮书必须包含“系统真值”章节，用来区分：

- 代码层能力是否已经实现
- 真实飞书环境里是否真的只有一个 responder 在回

### 9.2 当前版的诊断规则

`runtime_diagnostics` 作为当前系统真值入口之一，至少要能回答：

- 当前 official owner 是谁
- 最近心跳是否存在
- 最近事件是否被目标实例接收
- 最近 responder 来源是什么

### 9.3 验真期策略

- 验真期保留 `instance_tag` 或 debug footer
- 真机稳定后，再把 debug 信息收回到诊断模式
- 任何“手机上看起来通了”的判断，都要允许被 runtime 诊断推翻

### 9.4 当前版本已知事实

当前版本必须诚实写出：

- 代码层面的诊断与单 owner 支撑已经完成
- 真实飞书环境里是否只剩一个 responder，仍属于 rollout 验真项
- 因此“架构成立”和“live owner 完成切换”是两个不同层级的完成度

## 10. 当前版本的推荐操作法

### 10.1 手机端默认动作

当你在路上、在碎片时间、或不想打开 PC 时：

1. 先在飞书里发起
2. 优先用 `原力OS` 做：
   - 接任务
   - 续上下文
   - 看任务与今日重点
   - 轻记录
   - 轻查资料
   - 收口准备
3. 看到需要深工、长链执行、正式写作或代码时，再交给 PC

### 10.2 PC 端默认动作

打开 PC 后，默认把下列工作交给 `AI大管家`：

- 深度研究
- 复杂项目推进
- 代码与文件修改
- 批量处理
- 多工具编排
- 证据闭环与镜像同步

### 10.3 一天最顺的协同节奏

建议你把一天压缩成这条主链：

`今天最该看什么 -> 我现在有哪些任务 -> 原力原力 记一下 -> 原力原力 搜一下 -> 把这事闭环 -> 交给 PC 继续`

这条主链的设计目的不是让飞书完成所有工作，而是：

- 让你始终知道当前最正确的下一步
- 让上下文不要因为设备切换而断掉
- 让证据与闭环回到后台 canonical

## Assumptions

- 默认场景是你个人长期自用，中文，移动端高频，PC 端深工补位
- 默认 `原力OS` 继续只是飞书前台，不扩成独立后台
- 默认 `AI大管家` 继续是唯一治理核、唯一闭环 owner、唯一 canonical source
- 默认 Get笔记 v1 继续只做读路径参考，不把外部笔记写入作为前台默认能力
- 默认本版先固化协同协议，不扩 Siri、快捷指令和多人协作
- 默认真实飞书环境的“旧 responder 清理”属于 rollout 工作，不改变这份白皮书的系统设计
