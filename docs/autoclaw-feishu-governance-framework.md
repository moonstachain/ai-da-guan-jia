# AutoClaw / 飞书机器人 / AI大管家 统筹框架

这份文档不是一次性的调优说明，而是后续评估各种 `Claw` 类前端时都能复用的判断框架。

## 核心结论

- `AutoClaw` 不是飞书机器人的替代品，而是更强的个人前台壳。
- `飞书机器人` 不是 AutoClaw 的低配版，而是更强的组织协作入口。
- `AI大管家` 不是执行器，而是位于两者之上的治理层和总控层。

## 六个判断

- `自治判断`
  - AutoClaw 自用版可以高自治承接模糊任务入口与结构化判断；登录、授权、发布、支付、删除、审批拍板仍是人类边界。
- `全局最优判断`
  - 最优解不是二选一，而是 `AutoClaw 做个人前台总控壳 + 飞书机器人做协作和结果分发表面 + AI大管家做治理层`。
- `能力复用判断`
  - 先复用已经验证过的强收敛 keep set、冲突治理和 7 天评估机制，不把“换个前端”误当“要重做系统”。
- `验真判断`
  - 成功标准不是 skill 多，而是你是否更少纠结入口、更少遇到冲突、更容易形成稳定短链。
- `进化判断`
  - 新的 Claw 类前端一律按 `前台壳 / 治理层 / 执行层` 三层去看，避免再次把新壳误当全能系统。
- `当前最大失真`
  - 把“装了很多 skill”误判成“系统更强”，是最常见失真。

## AutoClaw vs 飞书机器人

| 维度 | AutoClaw | 飞书机器人 | AI大管家治理结论 |
| --- | --- | --- | --- |
| 入口形态 | 个人桌面工作台，适合单人连续对话和模糊任务入口 | 组织消息流中的机器人入口，适合消息触发、提醒和结果分发 | 个人先走 AutoClaw，协作再走飞书 |
| 用户心智负担 | 装好后更像个人工作台，但 skill 过多时容易失真 | 入口熟悉，但聊天噪音重，容易把消息往来误当闭环 | AutoClaw 适合先想清楚，飞书适合把结果送出去 |
| 协作能力 | 弱于飞书，不擅长多人消息、权限、审批和组织内事件驱动 | 强于 AutoClaw，天然在群聊、通知和审批边界里 | 多人协作、提醒、审批时飞书优先 |
| 任务路由能力 | 更适合承接模糊任务、自然语言分流和个人工作流起点 | 更适合明确触发，不适合承担复杂多 skill 冲突治理 | 任务还糊时先走 AutoClaw |
| 可控性与治理成本 | 不收敛 skill 时治理成本高；收敛后适合做个人总控壳 | 组织边界清晰，但复杂任务理解容易把治理成本藏在消息流里 | AI大管家要把两边职责切开 |
| 适合承接的任务类型 | 模糊任务入口、该不该做、先问哪个源、能不能收口 | 通知、提醒、把结果送到人和群、多人协作触发 | AutoClaw 负责想清楚，飞书负责送出去 |

## 四方分工

| 对象 | 定位 | 该做 | 不该做 | 停手点 |
| --- | --- | --- | --- | --- |
| AutoClaw | 个人前台总控壳 | 接模糊任务、做判断、选知识源、判断能否收口 | 不抢组织协作、消息分发、审批流，也不默认扮演全能执行层 | 一旦进入多人协作、组织通知或审批边界，应切到飞书或交回人 |
| 飞书机器人 | 协作和结果分发表面 | 通知、提醒、把结果送到人和群、承接多人协作触发 | 不承担个人任务理解、复杂路由和多 skill 冲突治理 | 一旦需要任务澄清、选技能链或做闭环判断，应交给 AI大管家统筹或回到 AutoClaw |
| AI大管家 | 治理层和总控层 | 整体盘点、优势劣势解释、前端壳选择、skill 链选择、边界治理、验真闭环、进化记录 | 不直接替代所有执行层 skill，也不把前台壳误当 canonical memory | 完成壳层判断、路由和闭环后，应把明确执行交给下游 skill 或把不可逆动作交给人 |
| 人 | 不可替代的最终决策者 | 登录、授权、发布、支付、删除、审批拍板、给出主观偏好、决定是否扩大条件启用集 | 不要手工在 96 个 skill 里反复挑工具，也不要把消息回复误判成真正完成 | 凡是不可逆或强主观动作都由人收口 |

## 前台使用原则

- 任务还糊、需要先压清楚推进链时，先走 AutoClaw。
- 问题本质是“该不该做”“先查哪个源”“这事能不能收口”时，优先把 AutoClaw 当入口。
- 任务已经是明确执行动作时，直接去下游执行层 skill，不让 AutoClaw 重复抢活。
- 结果需要发到人和群、进入多人协作、提醒或审批边界时，再切到飞书机器人。

## 治理原则

- 先判断任务应落在哪个前台壳，再判断是否需要下游 skill，再判断是否需要人拍板，最后判断是否真的完成。
- 不把聊天成功当完成；完成必须带证据、边界说明和开放问题。
- 不把飞书消息当 canonical memory；本地工件仍是真相源。
- 不把 skill 数量当能力；默认维持强收敛，只有验证后才扩大条件启用集。

## 未来更多 Claw 的通用问题

- 它更适合做前台壳、治理层，还是执行层？
- 它是否重复了现有 AutoClaw 或飞书机器人的职责？
- 它会降低入口心智负担，还是只是增加新的配置面和技能冲突？
- 接入它之后，AI大管家是否还能保持统一边界和验真闭环？
