# Or Force OS Miaoda Cockpit v1

这份文档定义 Miaoda 这页应该承担的角色：它是原力OS 的前台驾驶舱，不是治理本体。

## 页面定位

Miaoda 页面负责：

- 让人快速看懂系统当前状态
- 让人快速看到阻塞、风险和下一步动作
- 让人快速进入对应工作台或回写动作

Miaoda 页面不负责：

- 真相存储
- 复杂业务计算
- 规则最终裁决
- 取代 Feishu 表结构

## 页面结构

### 左侧：工作台入口

- 工作流入口
- 插件入口
- 任务入口
- 证据入口
- Feishu / GitHub 跳转入口

### 右侧：驾驶舱内容

#### 顶层四卡

- `健康度`
- `当前聚焦`
- `阻塞与风险`
- `本周进化`

#### 中层三块

- `运行态`
- `clone 态`
- `治理态`

#### 底层收口区

- 需要人类点击的事项
- 需要审批的事项
- 需要回写飞书的事项

## 视觉原则

- 深色、安静、偏控制台
- 一屏先看状态，再看动作
- 少噪音，少装饰，多信号
- 不做泛 BI 仪表盘感
- 不把设置项摆成首页主角

## 信息优先级

1. 当前是否健康
2. 当前在推什么
3. 哪里卡住了
4. 需要谁来点一下
5. 本周进化了什么

## 当前页面的落实方式

你现在已经有一个可工作的 Miaoda 页面骨架，因此实现方式不是重建，而是：

- 把现有页面的模块语义固定下来
- 把现有总控驾驶舱内容对齐到四卡 + 三块 + 收口区
- 把左侧工作台和右侧驾驶舱之间的跳转边界固定下来
- 把 `preview / source view / binding` 这些概念分清楚

## 与数据层的连接

Miaoda 读的不是“随手看到的文案”，而是 Feishu 多维表里已经结构化的 source view。

页面上的每个卡片都应该只回答一个管理问题。
