# 原力OS 协同治理仪表盘 Review Memo v2

## 结论

这次外部 benchmark 给我们的启发非常明确：

`原力OS` 下一步最该做的，不是“再漂亮一点”，而是从 `好看的治理仪表盘` 升级成 `可信的协同治理控制台`。

参考项目：[openclaw-control-center](https://github.com/TianyiDataScience/openclaw-control-center)

它最值得借的不是 OpenClaw 运行时，而是它对控制中心的理解：

- 先回答“系统现在整体正常吗”
- 再回答“如果不正常，我该先介入哪儿”
- 最后才回答“背后的运行细节是什么”

这和 `原力OS` 当前方向是高度同向的。  
所以 v3 不建议推倒重来，而是建议在 v2 的基础上做一次 `控制台化升级`。

## 这次最值得借的东西

### 1. 控制面姿态

外部项目把自己定义成一个 `安全优先、本地优先控制中心`，这很重要。  
它不是“信息展示页”，而是一个带有清晰边界和控制语义的操作台。

对 `原力OS` 的启发是：

- Hero 不能只说明“这是什么”
- Hero 还要说明“当前运行态是什么、风险态是什么、我是否需要介入”
- 页面语气应该更像系统操作界面，而不是概念海报

### 2. 真实工作与排队待命的区分

外部项目很强调：

- 谁正在真实执行
- 谁只是 backlog 排队
- 哪些东西只是看板映射，不是运行证据

这对我们很关键，因为 `原力OS` 本身就在解决真假闭环、真假推进、真假阻塞的问题。

对 v3 的直接要求是：

- 前台主线卡不能只展示“有哪些主线”
- 还要区分 `正在推进`、`只是在排队`、`已经停住`
- 任务状态和行动层必须带证据感，而不是只有漂亮标签

### 3. 证据与动作放在一起

外部项目把任务、审批、执行链和运行证据放在同一个控制面里看。  
这点非常值得我们借，因为 `原力OS` 不该停在“知道发生了什么”，而应该继续支持“知道为什么发生、我该如何介入”。

对 v3 的要求是：

- 诊断卡必须能引出具体行动
- 行动卡必须能回到具体对象和证据入口
- 不让 `blocked_system` 和 `blocked_needs_user` 继续停留在抽象状态名

### 4. 二级工作台入口

外部项目虽然是多分区后台，但它提醒我们一件很有价值的事：

单页总控舱要成立，背后必须有明确的二级工作台入口。

所以 v3 建议保留 `单页总控舱`，但要新增轻量入口，至少能引导到：

- 线程工作台
- 任务工作台
- 决策与写回
- 数据源同步

## 这次不要直接借的东西

### 1. 不直接借 OpenClaw runtime 结构

这个项目本质上还是 OpenClaw 控制中心。  
我们的对象是 `原力OS / AI大管家 / 协同治理`，治理语义更强，不能退化成 agent runtime 面板。

### 2. 不直接借多分区后台信息架构

第一页仍然应该保持：

- 单页
- 先 onboarding
- 再运行态
- 再诊断
- 再行动

不能因为参考项目做得成熟，就把 `原力OS` 首页直接展开成管理后台。

### 3. 不把运维信号抬成第一优先级

外部项目里 `用量 / 配额 / cron / live execution` 很重要。  
但对我们当前阶段来说，更高优先级仍然是：

- 当前主会场是什么
- 哪些是真阻塞
- 哪些需要人类介入
- 下一步最该做什么

### 4. 不把设置和配置前置

设置、权限、连接状态当然重要，但 v3 首页不该先把人带进配置视角。  
这些内容更适合作为二级工作台或边缘信息。

## v3 应该怎么升级

### 1. 从“人类看得懂”升级到“人类敢判断”

v2 解决的是 onboarding 缺位。  
v3 要继续解决的是：看懂以后，人类是不是知道现在该不该介入、为什么介入、先介入哪一个。

### 2. Hero 升级成控制态头部

Hero 建议新增这些信号：

- 当前运行态
- 当前 frontstage focus
- 风险级别
- 只读 / 治理模式说明

其中“只读 / 安全姿态”在 v3 可以先作为静态控制文案，不要求强绑定真实字段。

### 3. KPI 和诊断层更像控制中心

保留原有 4 KPI、4 诊断卡，但文案和视觉语气都要更偏控制台：

- 少一点“字段展示”
- 多一点“系统判断”

### 4. 行动层升级成介入控制面

行动层要从“列出队列”升级成“帮助人做决定”：

- `待你处理` 是必须介入的第一队列
- `系统阻塞` 是必须单列处理的异常面
- `下一步动作` 必须更明确
- `能力与审批摩擦` 继续保留，但要更像治理约束卡

### 5. 新增工作台入口

单页中建议增加轻量入口条或 utility rail，给出：

- `线程`
- `任务`
- `决策`
- `数据源`
- `文档 / 记忆`

这不是把首页做成后台，而是让总控舱不成为死页面。

## v3 的置顶项目判断

我建议把下一轮迭代项目直接命名为：

`原力OS 协同治理总控舱 v3：从漂亮仪表盘升级为可信控制台`

它的优先级顺序应该固定为：

1. `控制态更强`
2. `介入更精确`
3. `证据链更清楚`
4. `工作台入口更明确`
5. `视觉更贵、更安静`

而不是：

1. 继续加卡片
2. 继续加炫光
3. 继续把白皮书塞进页面

## 一句话判断

这个外部项目足够好，已经可以作为我们下一轮的置顶 benchmark。  
但它对我们的价值不是“照着做一个 OpenClaw 控制台”，而是帮助 `原力OS` 更坚定地走向：

`单页总控舱 + 控制台语义 + 证据驱动 + 人类可介入`
