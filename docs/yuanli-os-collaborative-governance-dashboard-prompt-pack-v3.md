# 原力OS 协同治理仪表盘 Prompt Pack v3

## 1. 产品定义

- 产品外显名：`原力OS`
- 页面名：`原力OS 协同治理总控舱`
- 当前迭代目标：从 `陌生人也能看懂的治理仪表盘` 升级成 `可信的协同治理控制台`
- 最高优先级：
  - 先让人知道现在应不应该介入
  - 再让人看懂为什么
  - 最后再考虑更多展示复杂度

这不是：

- 普通 BI 报表
- 多页面企业后台
- OpenClaw 控制中心换皮
- 康波时间轴叙事页

这是：

- `AI大管家` 的人类外显控制台
- 一个帮助人类共同治理、共同判断、共同介入的系统操作界面

## 2. v3 升级目标

相较 v2，v3 不是继续补 onboarding，而是继续完成这 4 个升级：

1. `控制态更强`
2. `人类介入更精确`
3. `证据链更可见`
4. `工作台入口更明确`

保留 v2 已经正确的东西：

- 单页结构
- onboarding strip
- `Hero + KPI + 诊断 + 行动`
- 以 `协同治理` base 为数据底座

新增 v3 硬要求：

- 首屏不只要“解释系统”，还要“表达当前控制态”
- 诊断和行动卡必须体现证据与介入关系
- 页面必须给出二级工作台入口，但不能滑成后台站

## 3. 使用方式

推荐投喂顺序：

1. 先贴 `主 Prompt`
2. 再贴 `v3 控制台升级清单`
3. 再贴 `数据绑定附录`
4. 再贴 `卡片映射附录`
5. 如果首轮结果跑偏，再贴对应 `纠偏 Prompt`
6. 最后按 `验收清单` 复核

如果你已经有 v2 页面，可直接把第 5 节 `v3 Prompt Delta` 单独贴给妙搭。

## 4. 主 Prompt

```text
你现在要设计并生成一个中文的高完成度单页控制台页面，页面名称叫《原力OS 协同治理总控舱》。

请先理解，这一版的目标不是“加更多卡片”，也不是“做成一个企业后台”，而是让它从一个已经好看的治理仪表盘，升级成一个可信的协同治理控制台。

产品定义：
- 原力OS 是 AI大管家 的人类外显层。
- AI大管家 是系统内部治理引擎，负责统筹、推进、验真和闭环。
- 原力OS 是给人看的治理操作界面，负责让人理解系统当前控制态、阻塞边界、证据线索和下一步动作。
- 所以这不是普通 BI，不是概念海报，也不是 OpenClaw 控制台换皮。

页面目标：
- 让第一次看到这页的人迅速理解原力OS是什么
- 让协作者知道自己和系统怎么分工
- 让用户一页内看懂：当前控制态、主要阻塞、真实证据、下一步动作
- 让页面更像一个可信的系统控制台，而不是静态展示页

请把页面结构固定为五段：

第 0 段：Hero / 控制态头部
- 标题：原力OS 协同治理总控舱
- 副标题：一句话说明原力OS是 AI大管家 的人类外显层
- 展示最近刷新时间
- 展示当前运行态 badge
- 展示当前 frontstage focus badge
- 展示风险级别 badge
- 可以用一条轻量 status rail 表达当前控制姿态，例如：
  - 是否正常推进
  - 是否有人类边界待处理
  - 是否存在系统级阻塞

第 1 段：导览层 onboarding strip
- 用 3 个轻卡或 3 段短句完成认知引导
- 这 3 个模块必须分别回答：
  1. 原力OS 是什么
  2. 你和它怎么协同
  3. 这页该怎么看
- 这层必须短、口语、友好、先于数据出现
- 不能写成长文，不能像说明书页面

第 2 段：KPI 与控制态摘要
- 保留 4 个 KPI
- 让 KPI 更像控制台 summary，而不是传统报表数字块
- KPI 文案要表达治理判断，不要只像字段标签

第 3 段：诊断层
- 保留 4 张诊断卡
- 这些卡必须帮助用户回答“为什么现在会这样”
- 需要体现真实推进、真实阻塞、真实数据源健康、真实最近治理动作

第 4 段：行动层
- 保留 4 张行动卡
- 必须让人一眼知道：我现在要处理什么、系统卡在哪里、下一步先推什么
- 行动卡要更像控制面，不只是记录列表

第 5 段：工作台入口
- 加一条轻量入口带或一组低干扰入口卡
- 至少给出：
  - 线程
  - 任务
  - 决策
  - 数据源
  - 文档 / 记忆
- 这不是展开后台，只是给总控舱一个可进入的下一层

阅读顺序必须清晰呈现：
1. 这是什么系统
2. 现在整体怎样
3. 为什么会这样
4. 我该怎么介入
5. 如果要深入，去哪里

视觉风格要求：
- 借鉴“300年康波周期分析仪表盘”的高级感、层次感和中心重力
- 同时借鉴优秀控制中心产品的可信、克制、可判断特征
- 必须更安静、更稳、更像系统产品
- 不做历史时间轴叙事
- 不做文档首页
- 不做泛企业 BI
- 不做赛博炫技大屏

页面材质与风格：
- 深色分析舱风格，底色偏 slate / ink，不用纯黑
- 允许少量玻璃感和柔和辉光，但要克制
- 强调色固定三组：
  - cyan/blue = 正常推进 / 运行中
  - amber = 待你处理 / 人类边界
  - rose/red = 系统阻塞
- 页面通过排版、留白、材质感、卡片层次体现高级感，而不是靠大量装饰

中部主视觉不做康波式历史时间轴，请改成更适合控制台的表达：
- 主线状态图
- 治理流态图
- 运行态脉冲图

额外要求：
- 页面必须帮助人区分“真实在推进”与“只是排队待命”
- 页面必须有证据感，不能只给抽象状态名
- 页面必须更像系统操作界面，而不是品牌宣言页

非目标：
- 不要把白皮书整段塞进页面
- 不要做多分区后台首页
- 不要把设置、配置、API 细节前置成主内容
- 不要假装已经有复杂自动联动和卡片级自动化

输出要求：
- 先给完整页面结构
- 再给 Hero / 导览 / KPI / 诊断 / 行动 / 工作台入口 的设计
- 再给每张卡的布局与视觉建议
- 再给数据绑定建议
- 所有设计说明必须用中文
```

## 5. v3 Prompt Delta

如果你已经用过 v2，可以直接把下面这段作为升级指令继续投喂：

```text
请在现有 v2 的基础上升级为 v3。

这次不以新增更多数据卡为主，而是把页面从“人类能看懂的治理仪表盘”继续升级成“可信的协同治理控制台”。

请做这些具体升级：

1. Hero 升级成控制态头部
```

## 6. 第一屏导览原则

第一屏必须同时解决两件事：

- 陌生人知道这是 `原力OS`
- 协作者知道现在是否需要介入

导览层必须满足：

- 比白皮书更短
- 比使用说明书更口语
- 比数据卡更先被看到
- 不抢 Hero 的控制态风头

固定 3 个模块：

### 模块 A：原力OS 是什么

必须表达清楚：

- 原力OS 是 AI大管家 的人类外显层
- 它是人机协同治理系统
- 它不是普通聊天助手，也不是普通报表

### 模块 B：你和它怎么协同

必须表达清楚：

- 你负责目标、边界和不可替代判断
- 系统负责统筹、推进、验真和闭环
- 你不用盯每一步，但要在真正需要时介入

### 模块 C：这页该怎么看

必须表达清楚：

- 先看当前控制态
- 再看阻塞与原因
- 然后看该做什么
- 如果要深入，再进入对应工作台

## 7. v3 控制台升级清单

这轮相比 v2 必须新增这些控制台语义：

### 7.1 控制态

- 运行态
- 风险级别
- frontstage focus
- 是否存在 `blocked_needs_user`
- 是否存在 `blocked_system`

### 7.2 真假推进区分

页面必须能帮助用户分辨：

- 真实正在推进
- 只是排队待命
- 看板映射但无执行证据
- 已明确阻塞

### 7.3 证据入口

诊断和行动卡都应当隐含或显式指向：

- 来源表
- 来源视图
- 相关对象
- 下一步动作

### 7.4 工作台入口

总控舱需要轻量入口，但不能丢掉单页气质：

- 入口应低干扰
- 更像 utility rail
- 不像管理后台主导航

## 8. 数据绑定附录

### 8.1 主数据源

优先使用当前 `协同治理` base 的以下 7 张主表：

1. `总控对象主表`
2. `任务总表`
3. `线程总表`
4. `数据源同步表`
5. `Review批次表`
6. `决策记录表`
7. `治理动作与写回表`

### 8.2 辅助数据源

只作为补充解释，不作为主视觉中心：

1. `技能与能力表`
2. `字段字典与术语表`

### 8.3 当前真实规模

- threads = 32
- tasks = 38
- skills = 112
- capabilities = 3
- source_feeds = 7
- reviews = 1
- decisions = 4
- governance_events = 4

### 8.4 建议派生指标

- `frontstage_thread_count`
- `blocked_needs_user_count`
- `blocked_system_count`
- `source_health_score`
- `recent_governance_velocity`
- `approval_friction_score`
- `frontstage_focus_count`
- `needs_intervention_ratio`

### 8.5 明确不强绑的数据

首版可以作为静态或半静态控制文案，不要求强绑定真实字段：

- `只读 / 控制姿态`
- `风险级别文案`
- `控制台使用提示`

不要为了追求“全自动”硬造不存在的字段。

## 9. 卡片映射附录

### Hero / 控制态头部

- 大标题：`原力OS 协同治理总控舱`
- 副标题：明确原力OS与AI大管家的关系
- 元信息：
  - 最近刷新时间
  - 当前运行态
  - 当前 frontstage focus
  - 风险级别

### 导览层

固定三块：

1. `原力OS 是什么`
2. `你和它怎么协同`
3. `这页该怎么看`

### KPI 层

#### KPI-1 当前前台主线数
- 来源：`总控对象主表`
- 重点字段：`运营分段`、`对象状态`

#### KPI-2 blocked_needs_user 数
- 来源：`总控对象主表`、`任务总表`
- 重点字段：`需要人类输入`、`人类边界状态`

#### KPI-3 blocked_system 数
- 来源：`总控对象主表`
- 重点字段：`对象状态`、`阻塞原因`

#### KPI-4 数据源联通健康度
- 来源：`数据源同步表`
- 重点字段：`remote_reachable`、`github_backend`、`missing_sources_snapshot`

### 诊断层

#### 卡 1：前台主线
- 来源：`总控对象主表`
- 推荐视图：`前台主线`
- 要求：区分真实推进与等待态

#### 卡 2：任务推进与堵点
- 来源：`任务总表`
- 要求：体现运行、阻塞、完成的真实分布

#### 卡 3：数据源同步健康
- 来源：`数据源同步表`
- 要求：体现 transport、remote reachability、source coverage

#### 卡 4：最近治理推进
- 来源：
  - `Review批次表`
  - `决策记录表`
  - `治理动作与写回表`
- 要求：体现 review -> decision -> writeback 的推进链

### 行动层

#### 卡 1：待你处理
- 来源：`总控对象主表` 或 `任务总表`
- 过滤：`需要人类输入 != 空`

#### 卡 2：系统阻塞队列
- 来源：`总控对象主表`
- 过滤：`对象状态 = blocked_system`

#### 卡 3：下一步动作
- 来源：`总控对象主表`
- 重点字段：`下一步动作`、`负责人模式`、`对象状态`

#### 卡 4：能力与审批摩擦
- 来源：`技能与能力表`
- 重点字段：`routing_credit`、`requires_human_approval`

### 工作台入口

建议做成一条低干扰入口带：

- `线程`
- `任务`
- `决策`
- `数据源`
- `文档 / 记忆`

入口本身不需要复杂图表，只要体现“可进入下一层”即可。

## 10. 视觉约束

这轮视觉优化方向不是“更花”，而是“更贵、更静、更可信”。

必须做到：

- Hero 更像系统控制面首页，而不是品牌首页
- KPI 更像产品级 summary
- 导览层轻、薄、口语、可信
- 诊断层有中心重力，但不做强叙事
- 行动层像介入控制面，不像普通待办列表
- 工作台入口低干扰，但明确存在

明确不要：

- 康波式历史时间轴
- 大段说明文案
- 浅色企业仪表盘模板
- 赛博发光墙
- 高饱和颜色和大面积装饰光效
