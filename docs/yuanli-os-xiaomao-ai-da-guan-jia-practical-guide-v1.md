## 1. 先说结论

当前版本已经可以作为你 `单人 / 高频 / 移动端` 的日常前台主入口来用。

但“可以彻底使用”的准确说法应该是：

- `原力OS`
  - 已经可以承担你日常飞书前台协同主入口
- `小猫`
  - 已经可以承担项目推进、跟盯催、节奏提醒
- `AI大管家`
  - 仍然是唯一后端治理核、唯一 canonical、唯一闭环 owner
- `不能误解的边界`
  - 飞书前端不是深工执行层
  - 飞书前端也不是“所有事情都能在聊天里做完”的总替代

一句话总结：

`可以日常使用，但不能把前端误当成万能后端。`

## 2. 四方角色到底怎么分

| 角色 | 你可以把它理解成什么 | 主要负责什么 | 不负责什么 |
| --- | --- | --- | --- |
| 人类 | 最终决策者 | 提目标、补偏好、做授权、做主观判断、做最终验收 | 不需要亲自挑技能、做系统路由、盯所有状态细节 |
| 原力OS | 飞书聊天前台 | 连续陪伴、轻路由、轻摘要、轻判断、轻记录、上下文恢复 | 不抢长链执行、不抢深研究、不抢不可逆动作 |
| 小猫 | 项目推进器官 | 汇总未完任务、识别停滞、跟盯催、提醒两边、提请拍板 | 不直接宣布完成、不替你做不可逆动作、不替代 PC 深工 |
| AI大管家 | 后端治理大脑 | route、执行、证据、状态、闭环、治理 | 不把聊天成功当完成，不把飞书前台当 canonical |

一句最关键的定位写死为：

- `原力OS` 是你聊天的前台
- `小猫` 是前后端之间的项目推进器官
- `AI大管家` 是真正做事、做真、做闭环的后端大脑

## 3. 怎么实际使用飞书前端

### 3.1 最常用的前台说法

| 你怎么说 | 这句话的用途 | 系统应该返回什么 |
| --- | --- | --- |
| `原力原力` | 看首页 / 可用入口 | 现在能做什么、下一步怎么开口 |
| `继续昨天那个` | 恢复上下文 | 上次推进到哪、当前 run、下一步 |
| `我现在有哪些任务` | 看任务总表 | 活跃任务、阻塞、待拍板、优先级 |
| `今天最该看什么` | 看今日重点 | 今日最值得看的任务和下一步 |
| `小猫现在在推什么` | 看项目推进节奏 | 本小时总判断、最该推进的一件事、当前最大失真 |
| `原力原力 记一下 ...` | 轻记录 | 形成 canonical capture，不等于闭环 |
| `原力原力 搜一下 ...` | 轻知识查询 | 先查路径，再给最小可证结论 |
| `把这事闭环` | 收口准备 | ready / not_ready / blocked_* |
| `这件事交给 PC 继续` | 切回深工 | 为什么要切、接力点在哪 |

### 3.2 日常建议的使用顺序

1. 早上先问：
   - `今天最该看什么`
2. 再看推进节奏：
   - `小猫现在在推什么`
3. 白天碎片时间：
   - `我现在有哪些任务`
   - `原力原力 记一下 ...`
   - `原力原力 搜一下 ...`
4. 遇到断点时：
   - `继续昨天那个`
5. 临近收尾时：
   - `把这事闭环`
6. 需要深工时：
   - `这件事交给 PC 继续`

### 3.3 一个最小工作流

这条链路就是当前版最推荐的日常用法：

`今天最该看什么 -> 小猫现在在推什么 -> 我现在有哪些任务 -> 记一下 -> 搜一下 -> 把这事闭环 -> 交给 PC 继续`

如果只用一句话记住这套节奏：

`先在飞书发起，小猫盯推进，深工切 PC，闭环看证据。`

## 4. 怎么理解小猫

### 4.1 小猫到底是什么

小猫不是“另一个 AI大管家”，也不是“第二个后端大脑”。

它更准确的定位是：

- `项目推进器`
- `跟盯催层`
- `前后端节奏协调层`

也就是说，它的核心价值不是“替你做更多事”，而是：

- 不让任务悄悄停掉
- 不让前后端彼此失联
- 不让人类待拍板事项被埋掉
- 不让“表面还在动”掩盖“实际上已经停了”

### 4.2 小猫看的输入

小猫默认看这些输入：

- canonical 活跃任务
- frontdesk session / capture
- requested_frontdesk_state
- heartbeat scorecard
- action queue
- human-boundary queue

### 4.3 小猫产出的东西

它的输出分成两边：

- 对人类
  - 提醒
  - 催办
  - 待拍板提示
  - 小时推进简报
- 对 AI大管家
  - 把停滞任务重新推回治理链
  - 把 close readiness、follow-up、human boundary 重新提高优先级
  - 把“没真推进但看起来像在推进”的任务重新显性化

### 4.4 小猫明确不做什么

- 不直接宣布完成
- 不直接改 canonical task 状态
- 不直接替你做不可逆动作
- 不替代 PC 深工
- 不替代 AI大管家 的 route、执行、证据和闭环

## 5. 前后端怎么一起协同

### 5.1 原力OS 负责什么

原力OS 负责：

- 让你表达更顺
- 让入口更轻
- 让上下文不断
- 让你在手机上就能看、问、定、记、催、转

### 5.2 小猫 负责什么

小猫负责：

- 盯住所有还没彻底结束的任务
- 识别哪些在推进、哪些停滞、哪些卡在人、哪些卡在系统
- 提醒人类
- 提醒后端
- 让前后端保持一条持续推进链

### 5.3 AI大管家 负责什么

AI大管家 负责：

- 真正推进任务
- 选择技能链
- 落证据
- 维护状态
- 做闭环判断
- 给出真实结果，而不是安慰性结果

### 5.4 人类负责什么

人类负责：

- 提目标
- 选方向
- 做授权
- 做主观判断
- 做最终验收

## 6. 一天怎么配合最顺

### 6.1 早上

建议顺序：

1. `今天最该看什么`
2. `小猫现在在推什么`

目的：

- 先看全局最值得看的事
- 再看有没有任务已经开始停滞

### 6.2 白天碎片时间

最适合做：

- `原力原力 记一下 ...`
- `原力原力 搜一下 ...`
- `我现在有哪些任务`

目的：

- 用最小成本把灵感、线索、问题先挂住
- 不要求每条都当场闭环

### 6.3 推进中

最适合做：

- `继续昨天那个`
- `小猫现在在推什么`

目的：

- 快速恢复上下文
- 确认有没有“表面在动，实际停掉”的任务

### 6.4 临近收口

最适合做：

- `把这事闭环`

目的：

- 不是立刻宣布完成
- 而是先判断是否已经具备闭环证据

### 6.5 需要深工时

最适合做：

- `这件事交给 PC 继续`

目的：

- 把飞书前台和 PC 深工明确分层
- 避免把复杂执行强塞进聊天窗

## 7. 真闭环到底是什么意思

真闭环不是：

- “聊完了”
- “飞书里看起来很顺”
- “小猫已经提醒过了”
- “AI大管家 好像已经懂了”

真闭环必须同时满足：

1. 有目标
2. 有执行
3. 有证据
4. 有状态
5. 有人类边界说明

所以：

- `记一下` 不是闭环
- `搜一下` 也不是闭环
- `小猫现在在推什么` 更不是闭环
- `把这事闭环` 只是进入闭环判断入口

## 8. 现在可以怎么说，不能怎么说

### 8.1 可以这样说

- `飞书前端已经可以日常使用`
- `原力OS 已经能承担前台主入口`
- `小猫已经能承担项目推进层`
- `AI大管家 仍然是唯一后端治理核`

### 8.2 不能这样说

- `飞书已经替代 PC`
- `小猫已经等于 AI大管家`
- `聊天成功就等于闭环`
- `前端已经完全零风险 rollout`

## 9. 当前最推荐的心法

如果只记四句，记这四句：

1. `我先在飞书发起`
2. `小猫负责盯推进`
3. `需要深工再切 PC`
4. `一切以证据为准`

## 10. 当前版已知边界

当前版最真实的边界是：

- 前台已经能日常用
- 小猫已经能进入项目推进层
- 但前台仍然不是深工执行层
- rollout / mirror / automation 仍有少量系统级边角
- 所以当前状态应理解为：

`前台日常可用，不等于全系统零风险。`
