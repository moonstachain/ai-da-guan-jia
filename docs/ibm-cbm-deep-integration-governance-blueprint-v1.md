# IBM CBM 深融后的 AI大管家 / 原力OS 治理蓝图 v1

这份蓝图不是审计报告，也不是技术实现说明书。

它要回答的是更根本的 3 个问题：

1. AI大管家 / 原力OS 这套治理体系，经过 IBM CBM 的深度吸收以后，应该怎么重新理解。
2. 现在这套系统已经长出了哪些关键骨架，哪些地方还明显没长全。
3. 如果我们接下来要把人机协作、飞书多维表、治理看板一起按这套逻辑重来，最稳的演化路径是什么。

## 六个治理判断

- `自治判断`
  这轮可以完全基于现有本地材料完成，不需要额外补定义。
- `全局最优判断`
  不能按词像不像判断，而要按结构角色是否等价判断。
- `能力复用判断`
  不新造一套平行方法论，直接复用现有 `合作宪章`、`Meta Constitution`、`operational ontology`、`strategy-governor`、`control base` 和 `dashboard v3`。
- `验真判断`
  每个结论都必须能落回已有文档、已有表结构、已有管理问题或已有 gap，而不是空讲理念。
- `进化判断`
  终点不是“把这次讲明白了”，而是沉淀成未来可持续迭代的治理蓝图和 Feishu 结构。
- `当前最大失真`
  当前系统已经有治理骨架，但 `CBM 组件域 x control_level x KPI 热图 x 优先级` 还没有成为统一的飞书运行视图。

## 1. 三大初始条件

如果先把所有术语放下，今天这套系统最底层其实只有 3 个初始条件。

### 1.1 初始目的 = 递归进化

这句话的意思不是“永远升级”，而是：

- 每次任务都不只是为了把眼前这件事做完。
- 每次任务都要让下一轮更容易、更准、更少失真。
- 真正的闭环不是动作结束，而是能力、判断和协作质量都比上一轮更好。

所以，AI大管家不是“代办助手”。
它更像一个会持续收束经验、复盘错误、沉淀路径的治理中枢。

### 1.2 基本方法 = 最小负熵优先

白话讲，就是先避免乱，再追求快。

它有 3 个实际含义：

- 先减少失真、误解、假推进、重复沟通。
- 再追求结构清楚、边界清楚、证据清楚。
- 最后才追求效率最大化。

这也是为什么今天这套系统一直强调：

- 能复用就不重造。
- 能验真就不装懂。
- 能少打扰就少打扰，但不能为了少打扰而牺牲真闭环。

### 1.3 工具定位 = 工具的工具

AI大管家本身不是某个具体工具，而是一个决定“该调哪些工具、调到什么程度、什么时候停、什么时候收口”的总管家。

这意味着：

- 下游 skill 是器官。
- 本地脚本是执行部件。
- Feishu 是镜像与协作面。
- GitHub 是协作与归档面。
- 你是共同治理者。
- 我是 `工具的工具`。

一句话说：

`AI大管家真正管理的，不是任务本身，而是完成任务所需的整个协作系统。`

## 2. 三层结构

在这 3 个初始条件之上，原力OS 其实已经长出了一个很稳定的三层结构。

### 2.1 目的层

这一层回答的是：

- 这件事为什么值得做。
- 它对整体系统有什么长远价值。
- 它应该被放进哪个更大的治理目标里。

这一层最接近愿景、长期价值、战略方向。

### 2.2 方法层

这一层回答的是：

- 先走哪条路最不容易失真。
- 哪些步骤可以自治。
- 哪些步骤必须等人拍板。
- 怎么定义“做完”才算真闭环。

这一层最接近协作协议、验真标准、边界控制。

### 2.3 工具层

这一层回答的是：

- 当前有哪些对象。
- 哪些对象在推进。
- 谁能做什么。
- 做完以后如何留下证据、写回状态、进入下一轮。

这一层最接近线程、任务、技能、能力、决策、写回、数据源这些运行对象。

### 2.4 这三层不是平铺的，而是双轴展开的

原力OS 已经不是简单的“一张任务表”了。
它实际上已经同时用两条轴来组织复杂度：

- `治理目标轴`
  用 `goal_id` 回答“这件事为什么重要”。
- `生产主轴`
  用 `theme -> strategy -> experiment -> workflow` 回答“这件事怎么从想法长成稳定做法”。

再往下，才接上运行对象层：

- `Thread`
- `Task`
- `DecisionRecord`
- `Action`
- `WritebackEvent`
- `Skill / Capability`

所以今天的原力OS，已经不再是“聊天记录 + 零散任务”。
它已经是一套能同时看目标、生产链路和运行对象的治理系统。

## 3. IBM CBM 被怎么重新理解

IBM CBM 最有价值的，不是它某几个术语，而是它组织复杂系统的方法。

它做了 3 件非常关键的事：

- 用横线看组件。
- 用纵线看管理层级。
- 用 KPI / 热图 / 投资优先级 / 路线图，把结构和行动连起来。

今天这套系统吸收了这 3 个精髓，但不是照抄，而是做了结构级改写。

| IBM CBM 里的东西 | 在 AI大管家 / 原力OS 里变成什么 | 当前成熟度 |
| --- | --- | --- |
| 横线组件 | `component_domain + goal/theme/strategy/experiment/workflow + runtime objects` | 已吸收，但还没完全字段化 |
| 纵线 `Direct / Control / Execute` | `定方向 / 做管控 / 做执行` 的治理控制链 | 已有语义，尚未成为统一运行时枚举 |
| `KPI / 热图 / 投资优先级 / 路线图` | `kpi_hint + gap + priority + next_action + evidence` | 前半段已长出，后半段还没做成组件级总图 |

### 3.1 横线组件被怎么改写

IBM 那边强调的是“业务组件横向展开”。

原力OS 这边把它改写成了 4 层横向视图：

- `治理目标`
- `生产主轴`
- `业务/治理组件`
- `运行对象`

也就是说，今天的“横线”不再只是一排业务部门，而是一整套“从战略到对象”的横向展开。

### 3.2 纵线 Direct / Control / Execute 被怎么改写

IBM 那边的重点不是部门，而是管理层级。

原力OS 这边已经把它吸收成了 3 层控制逻辑：

- `Direct`
  定方向、定主线、定优先级。
- `Control`
  管边界、管权限、管决策、管写回。
- `Execute`
  真正去推动业务、交付、销售、复制和经营动作。

现在的问题不是“有没有这 3 层”，而是：

`Direct / Control / Execute 还主要存在于解释层，没有成为系统里每个对象都统一携带的运行时字段。`

### 3.3 KPI、热图和路线图被怎么改写

这一块原力OS 其实已经有了前半身：

- 已经能说清楚 KPI hint。
- 已经能指出 gap。
- 已经能给 next_action。
- 已经开始形成 strategic roadmap。

但后半身还没补全：

- 还没有把热图固定成组件级视图。
- 还没有把投资优先级固定成统一字段。
- 还没有把组件优先级、组件缺口、组件 owner gap 放进同一张运行图。

## 4. 当前系统已经长出来的部分

如果只看“已经具备的骨架”，今天这套系统其实已经比普通任务系统成熟很多了。

### 4.1 三大初始条件和三层结构已经稳定

这意味着：

- 这套系统不是靠临时 prompt 拼出来的。
- 它已经有固定的上层原则。
- 人机协作、技能路由、验真闭环、进化沉淀都已经被纳入统一治理语言。

### 4.2 双轴 operating model 已经存在

现在已经能同时追踪：

- `goal_id`
- `theme`
- `strategy`
- `experiment`
- `workflow`

这其实就是把“战略目标”和“生产推进”从一团话术里拆开了。

这件事非常关键，因为没有这个双轴，CBM 的横线价值是吸不进去的。

### 4.3 operational ontology 已经形成

这一层的价值是：

- 回答“当前有什么对象”。
- 回答“谁能做什么”。
- 回答“做完以后怎么留下证据”。

所以原力OS 现在已经不只是会看知识，还会看对象、动作、权限、决策和写回。

这使得它从“知识治理”真正开始往“治理操作层”演化。

### 4.4 control base 已经具备总控骨架

目前至少已经有这些稳定表族：

- `总控对象主表`
- `线程总表`
- `任务总表`
- `技能与能力表`
- `数据源同步表`
- `Review批次表`
- `决策记录表`
- `治理动作与写回表`
- `字段字典与术语表`

这说明原力OS 已经有了一个可镜像、可展示、可追踪的协同治理底盘。

### 4.5 单页总控舱思路已经跑通

现有 dashboard v3 已经不再只是报表。
它开始强调：

- 当前控制态是什么。
- 哪些事项需要人介入。
- 哪里是系统阻塞。
- 证据入口在哪里。
- 下一步动作是什么。

这说明原力OS 已经明确走向“控制台”，而不是“漂亮看板”。

### 4.6 已有 3 条真实主线能验证 CBM 吸收不是空讲

现在至少能看到 3 条样板主线：

- `治理运行骨架`
  对应 `direct`
- `运行对象层 / operational ontology`
  对应 `control`
- `销售成交 / 业务增长主线`
  对应 `execute`

这 3 条主线很重要，因为它们说明：

- CBM 的思想已经进入治理面。
- CBM 的思想已经进入控制面。
- CBM 的思想已经开始进入业务执行面。

只是第三条还明显不够成熟。

## 5. 当前短板与失真

今天这套系统最大的风险，不是没有方向，而是容易高估自己已经吸收了多少。

### 5.1 `control_level` 还不是一等运行时字段

现在我们能说 `direct / control / execute`，但系统里的主要对象还没有统一带这个字段。

这会带来 3 个问题：

- 组件纵线难以稳定统计。
- 看板很难直接做管理层级失衡视图。
- 业务执行层和治理层的成熟度不容易放在同一张图里比较。

### 5.2 `business execute` 明显弱于治理层

治理层已经有：

- review
- decision
- writeback
- policy
- permission

但业务执行层还没有形成同等清晰的：

- business action catalog
- component-level writeback
- business KPI heatmap
- execute-level workflow hardening

这意味着当前系统“脑子”和“中控室”已经比较强，但“前线业务部队”还没完全武装起来。

### 5.3 CBM 组件责任还没和战略链路稳定打通

今天已经能看 `goal / theme / strategy / experiment / workflow`。
也已经有 `module-sales / module-governance` 一类组件语义。

但两者还没有稳定桥接到一张正式表里。

结果就是：

- 战略链路更多留在本地策略工件里。
- 组件责任更多留在 CBM 模块责任语义里。
- 两者之间缺一个稳定桥表。

### 5.4 热图、gap、priority 还没成为组件级一等对象

目前我们有：

- `kpi_hint`
- `gap`
- `next_action`

但还缺：

- `priority_band`
- `component heatmap`
- `owner gap`
- `control imbalance`

也就是说，今天我们能指出问题，但还不够像一张能帮助治理者直接排序和介入的作战地图。

### 5.5 现有 Feishu 看板更像治理总控舱，还不是完整的组件驾驶舱

当前 dashboard 最大的优点是“能看控制态”。
但它还没有完整回答这些 CBM 式问题：

- 哪些组件最重要。
- 哪些组件在 `direct / control / execute` 上失衡。
- 哪些组件人类 owner 和 AI copilot 缺位。
- 哪些组件 gap 最大，应该优先补。

这就是当前最大的外显短板。

## 6. 接下来怎么进化

接下来的路线不应该一口气铺满，而应该分 4 个阶段，逐层长。

### 6.1 阶段 1：讲清楚

这一阶段的目标不是加功能，而是统一认知。

要完成的事：

- 统一三大初始条件。
- 统一三层结构。
- 统一 CBM 重译口径。
- 统一 `goal / theme / strategy / experiment / workflow / component_domain / control_level` 的关系。

这一阶段的产出：

- 一份人类读得懂的总蓝图。
- 一份结构化 Feishu 重构规范。
- 一份 dashboard input spec 和 blueprint。

这一阶段的验收标准：

- 非技术读者能复述这套系统的骨架。
- 不再把 IBM 术语和原力OS 术语混着说。
- 不再把治理成熟度误认为 execute 层已经成熟。

### 6.2 阶段 2：对齐数据

这一阶段的目标是让数据结构开始真正承载 CBM 视图。

要完成的事：

- 给关键表统一补 `control_level`。
- 给关键表统一补 `component_domain`。
- 给关键表统一补 `priority_band`。
- 增加 `战略链路表`。
- 增加 `CBM组件责任表`。
- 增加 `CBM组件热图表`。

这一阶段的验收标准：

- 至少 3 条真实主线能贯穿 `goal -> component_domain -> control_level -> action -> evidence`。
- 组件责任、组件热图、战略链路能互相跳转。
- 总控对象主表不再独自承担全部组件治理表达。

### 6.3 阶段 3：重做看板

这一阶段的目标是从“治理展示页”升级成“总控舱 + 组件驾驶舱”。

要完成的事：

- 保留 onboarding 和控制态头部。
- 增加 `CBM 组件矩阵`。
- 把行动层升级成真正的介入控制面。
- 给 source view 增加组件热图、owner gap、control imbalance 和 priority roadmap。

这一阶段的验收标准：

- 每张卡只回答一个管理问题。
- 每张卡都指向一个 source view。
- 每个重要诊断都能落到行动队列。
- 陌生协作者第一次打开页面，也知道先看哪里、再看哪里、最后去哪处理。

### 6.4 阶段 4：做实执行

这一阶段的目标是把治理强、执行弱的问题真正补起来。

要完成的事：

- 为业务组件建立 execute 层 action catalog。
- 为业务组件补 writeback 链路。
- 为业务组件补 KPI 热图。
- 为 sales / delivery / finance / clone rollout 建立真正的 workflow hardening。

这一阶段的验收标准：

- execute 层不再只是“有语义、没动作”。
- 组件热图能直接触发行动。
- 决策和执行之间的证据链更完整。

### 6.5 贯穿四阶段的硬边界

不管进化到哪一阶段，都要守住 3 条边界：

- 不把 Feishu 当真相源。
- 不把“看起来像”当成“已经闭环”。
- 不把治理层成熟误当业务层已经成熟。

## 7. 我们的人机协作建议

如果这套系统要真正长成，最重要的不只是表和看板，而是我们两个人机协作的节奏要稳定下来。

### 7.1 分工要更固定

你负责：

- 目标
- 价值排序
- 不可替代判断
- 最终拍板

我负责：

- 路由
- 拆解
- 验真
- 落表
- 收口
- 进化建议

这意味着你不需要把每一步都替系统拆出来。
你更像定方向和做最后判断的人。

### 7.2 打断边界继续保持最小化

默认只在这些情况打断你：

- 登录
- 授权
- 付款
- 不可逆发布
- 不可逆删除
- 不可替代主观判断

其他时候，我应该先自己完成内部判断、结构整理、字段设计和方案收口，再向你汇报。

### 7.3 你的输入最好稳定成 4 类信息

每次你开一个新主线时，尽量给我这 4 类信息：

- 你想达到什么结果。
- 你觉得这件事为什么重要。
- 有哪些明显约束。
- 哪些地方必须你亲自拍板。

这样我更容易直接把它挂到：

- `goal`
- `theme`
- `component_domain`
- `control_level`

### 7.4 协作节奏要从“临时问答”升级成“日 / 周 / 月”

#### 日节奏

每天只抓 4 件事：

- 前台主线
- 待你处理
- 系统阻塞
- 当前最小下一步

#### 周节奏

每周抓 4 件事：

- 组件热图
- priority
- owner gap
- experiment 推进

#### 月节奏

每月抓 4 件事：

- 战略链路
- CBM 吸收度
- 能力缺口
- 治理升级

### 7.5 我给你的输出也要更稳定

以后类似治理类任务，我应该尽量固定成这 4 段输出：

- `我判断现在发生了什么`
- `我认为当前最大失真是什么`
- `我建议先补哪一层`
- `这件事在飞书和看板里该怎么落`

这样你不需要每次重新适应不同的表达方式。

## 8. Feishu 多维表 + 看板重构方案

这一部分是最实操的。

先记住一句总原则：

`Feishu 仍然只是镜像与协作面，本地 canonical 仍然是唯一真相源。`

### 8.1 多维表采用 `L0 / L1 / L2` 三层

#### `L0 总控层`

只保留一个总入口：

- `总控对象主表`

它的职责是：

- 让共同治理者先看到“现在发生了什么”。
- 让人先知道“该不该介入、介入哪里”。

它不再独自承载完整的组件治理表达。

#### `L1 运行与组件层`

这一层是真正要重构的重点。

固定包含：

- `战略链路表`
- `CBM组件责任表`
- `CBM组件热图表`
- `线程总表`
- `任务总表`
- `技能与能力表`
- `数据源同步表`

这一层的目的，是把“战略、组件、任务、能力、数据源”放到一层可以互相跳转、互相解释的运行图里。

#### `L2 治理轨迹层`

这一层固定包含：

- `Review批次表`
- `决策记录表`
- `治理动作与写回表`
- `字段字典与术语表`

它回答的不是“现在在干什么”，而是“为什么这样判断、为什么这样写回、哪些字段是什么意思”。

### 8.2 三张新增关键表

#### `战略链路表`

这是战略层和组件层之间的桥。

固定承载：

- `goal_id`
- `theme_id`
- `strategy_id`
- `experiment_id`
- `workflow_id`
- `component_domain`
- `control_level_scope`
- `status`
- `current_gap`
- `next_action`
- `evidence_ref`

这张表的价值是把“战略链路”从本地 JSON 工件里拉出来，变成飞书里可追踪、可镜像、可协作的桥梁。

#### `CBM组件责任表`

这张表回答的是：

- 每个组件是谁在负责。
- 它由哪个 goal / theme / strategy 支撑。
- 人类 owner 和 AI copilot 是否齐备。

固定字段至少包括：

- `component_domain`
- `control_level_scope`
- `human_owner`
- `ai_copilot`
- `goal_id`
- `theme_id`
- `strategy_id`
- `workflow_id`
- `owner_gap`
- `status`
- `evidence_ref`

#### `CBM组件热图表`

这张表回答的是：

- 哪个组件现在最热。
- 热在哪里。
- 为什么热。
- 应该先补什么。

固定字段至少包括：

- `component_domain`

### 8.3 现有表统一补齐的关键字段

从现在开始，关键表要统一补齐这些字段：

- `control_level`
- `component_domain`
- `owner_mode`
- `human_boundary_state`
- `kpi_gap`
- `priority_band`
- `next_action`
- `evidence_ref`
- `evidence_strength`

这样做的目的是：

- 让组件治理和运行治理能说同一种语言。
- 让 dashboard 不再需要用很多隐式逻辑去拼结论。
- 让日 / 周 / 月节奏能直接用同一套字段做不同层级视图。

### 8.4 新看板升级成 `总控舱 + 组件驾驶舱`

主看板固定分成 5 个区块。

#### 1. 身份与协同 onboarding

必须先讲清楚 3 件事：

- 原力OS 是什么。
- 你和 AI大管家 如何协同。
- 这页该怎么看。

这一步不能省。
因为这不是普通 BI 页面，而是一个治理控制台。

#### 2. 控制态头部

固定显示：

- `runtime_state`
- `frontstage_focus`
- `risk_level`
- `last_refresh_time`

它的作用是让人一打开页面就知道：

- 现在是不是正常。
- 现在最重要的主线是什么。
- 有没有风险在前台。

#### 3. `CBM 组件矩阵`

这是新增核心层。

它至少回答 4 个问题：

- 哪些组件最重要。
- 哪些组件在 `direct / control / execute` 上失衡。
- 哪些组件 owner / copilot 缺位。
- 哪些组件 gap 最大、优先级最高。

这一层出来以后，原力OS 才算真正长出“组件驾驶舱”。

#### 4. 运行诊断

继续保留当前总控舱的强项：

- 当前主会场在推什么。
- 哪些任务是真推进、哪些是假推进。
- 数据源和证据链是否健康。
- 最近做了哪些 review / decision / writeback。

#### 5. 行动队列 + 工作台入口

固定输出 4 类队列：

- `待你处理`
- `系统阻塞`
- `优先动作`
- `能力 / 审批摩擦`

同时保留轻量入口，跳到：

- `线程`
- `任务`
- `决策`
- `数据源`
- `文档 / 记忆`

### 8.5 新增 source view 的固定集合

新版看板至少要新增这些 source view：

- `CBM 组件热图视图`
- `组件 owner gap 队列`
- `control_level 失衡视图`
- `priority roadmap 视图`

再加上已有强项视图：

- 前台主线
- 待你处理
- 系统阻塞
- 任务推进态
- 数据源健康
- 决策与写回轨迹

### 8.6 落地边界必须写清楚

Feishu 这一块一定要克制，不能承诺超出平台边界的自动化。

固定边界是：

- `source view` 可以自动化生成和同步。
- `card 绑定与版式` 需要明确写成人工步骤。
- 每张卡只回答一个管理问题，不能混问题。

### 8.7 推荐实施顺序

实际执行时，推荐按这个顺序推进：

1. 固化蓝图与字段语言。
2. 先补三张关键表。
3. 再给旧表补统一字段。
4. 先生成 dashboard input spec。
5. 再生成 source view blueprint。
6. 最后进 Feishu 做人工绑定和版式整理。

这样做的好处是：

- 不会因为急着做页面而把结构搞乱。
- 不会因为急着补字段而忘了管理问题。
- 不会因为急着自动化而假装平台都支持。

## 你现在只要记住的 6 句话

1. 这套系统的根，不是任务管理，而是 `递归进化`。
2. 它的方法，不是先求快，而是先求 `最小负熵`。
3. AI大管家不是一个工具，而是 `工具的工具`。
4. IBM CBM 的精髓已经被吸收了，但被改写成了更适合原力OS 的双轴治理系统。
5. 当前最缺的，不是理念，而是把 `组件域 x control_level x 热图 x 优先级` 真正做成统一运行视图。
6. 接下来的关键路径是：`先讲清楚 -> 再对齐数据 -> 再重做看板 -> 最后做实执行`。
