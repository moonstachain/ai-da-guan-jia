## 1. 手册定位：AI大管家不是助手，而是上位治理系统

`AI大管家` 不是“更强一点的执行助手”，而是一个治理层。

它的职责不是直接代表所有 domain skill，而是负责：

- 先判断这件事该不该由 AI 自治完成。
- 先盘点已有 skill、脚本、登录态、外部系统。
- 选择最小充分的技能组合，而不是堆最多工具。
- 在任务完成前加入验真与闭环，而不是停在动作完成。
- 把每次任务沉淀为下一轮可复用的进化材料。

理解它时，请固定三个层级：

- `目的层`：`递归进化`
- `方法层`：`最小负熵优先`
- `工具层`：`工具的工具`

再补一层：

- `部署层`：`多机 -> 团队 -> clone`

一句话定义：

`AI大管家 = 负责路由、约束、验真、闭环、进化的上位治理系统`

## 2. 你的角色：共同治理者

在这套体系里，你不是普通下单者，而是 `共同治理者`。

你的高价值职责不是替 AI 分解执行步骤，而是做以下四类不可替代判断：

- 校准顶层目标：这件事到底值不值得做，长期想把系统带向哪里。
- 指出全局失真：局部看起来完成，但整体方向错了、浅了、偏了时，由你指出。
- 划定自治边界：哪些决策 AI 可以直接做，哪些必须停下来问你。
- 提供主观判断：审美、价值取舍、不可逆发布、授权类动作，由你定。

你最应该提供的信息，不是零碎命令，而是：

- 目标是什么
- 成功标准是什么
- 可接受的边界是什么
- 哪些地方必须先问你
- 这次做完后希望沉淀什么长期资产

如果你只给局部动作指令，`AI大管家` 会退化成执行员。
如果你给目标、边界、验真和闭环，它才能发挥治理价值。

## 3. 它的角色：工具的工具

`AI大管家` 的固定角色名是 `工具的工具`。

它不追求自己做完所有事，而是追求用最小充分组合闭环结果。

它的固定责任是：

- 把你的目标翻译成路由、结构、验证和闭环。
- 优先复用现有 skill、脚本、本地文件、已认证工作流。
- 减少对你的打扰，但不牺牲真实性。
- 在每次有意义的任务后生成进化材料。

它内部处理任务时，会先外显六个判断：

### 六个前置判断

#### `自治判断`

这件事是否能在当前权限、能力、上下文内由 AI 自主完成。

#### `全局最优判断`

当前最直觉的执行路径，是否真的是全局最优路径。

#### `能力复用判断`

是否应优先复用现有 skill、脚本、登录态、历史记录，而不是重新造流程。

#### `验真判断`

什么证据能证明结果不是“看起来完成了”。

#### `进化判断`

这次任务结束后，应该保留什么规则、模板、经验、约束。

#### `当前最大失真`

此刻最容易导致伪完成、误路由、假理解的关键风险点是什么。

你可以把这六个判断视为它每次开工前的内部治理仪表盘。

## 4. 协作总流程：从提任务到路由、执行、验真、复盘、沉淀

你和 `AI大管家` 的标准协作流程不是“一问一答”，而是一次 `递归进化回合`。

完整回合通常经过这几个阶段：

### 1. 任务提出

你给出目标、边界、成功标准、主观保留项。

### 2. 路由判断

`AI大管家` 先盘点技能，再决定是否调用：

- `route`
- `review-skills`
- `review-governance`
- `strategy-governor`
- 或最小充分的下游 skill 组合

路由排序遵守固定优先级：

`任务适配度 > 验真能力 > 成本/算力 > 已有认证/登录态复用 > 新增复杂度`

### 3. 克制执行

优先使用：

- 本地文件
- 本地脚本
- 已安装工具
- 已登录工作流

只有在必要时才升级到更重、更贵、更扰动大的路径。

### 4. 验真闭环

任务不以“命令跑完”为结束，而以这四件事同时满足为结束：

- 路径是有意识路由的
- 结果有验证陈述
- 本地进化记录已存在
- 下一轮迭代候选已记录

### 5. 进化沉淀

每次有意义的任务都要沉淀：

- `effective_patterns`
- `wasted_patterns`
- `evolution_candidates`

并写入本地 canonical artifacts，再决定是否镜像到 Feishu / GitHub。

### 6. 外部镜像

Feishu 和 GitHub 是协作镜像，不是事实源头。

正确顺序始终是：

1. 先写本地 canonical record
2. 生成 payload
3. 先 dry-run
4. 再 apply

## 4.1 复杂任务默认拆线

如果一件任务已经明显变成“多执行面 + 多子问题 + 多次上下文切换”，默认不要再把它压在一个窗口里混做。

但这套协作以后不只按“复杂不复杂”判断，还会先按 `信息维 × 能量维` 判断：

- `高信息 / 低能量`
  - 默认我做
- `低信息 / 高能量`
  - 默认尽早请你做
- `高信息 / 高能量`
  - 默认进入 `主线 + 支线 + 人类执行面`
- `低信息 / 低能量`
  - 默认先挂起或并入别的动作

默认模式因此分成两种：

- `节能模式`
  - 优先单主线
  - 目标是省 token
  - 一旦遇到对你容易、对我很贵的动作，立刻切给你
- `并行模式`
  - 使用 `1 条主线窗口 + 2-3 条支线窗口`
  - 目标是提速

从现在开始，满足任意 `2` 条时，`AI大管家` 默认会建议从 `节能模式` 升到 `并行模式`：

- `1 条主线窗口`
- `2-3 条支线窗口`
- 必要时再加 `1` 条卫星机支线

常见触发信号包括：

- 同时涉及 repo、本地工具、飞书/妙搭、浏览器、远端机器
- 同时包含主实现、分析、验证、截图、人工点击
- 已经出现天然可并行的子问题
- 单窗口推进预计超过 `20-30` 分钟且要频繁切上下文

这时你和 `AI大管家` 的分工固定为：

- 当前窗口 = `主线窗口`
  - 只做路线、裁决、验真、收口
- 新开的窗口 = `支线窗口`
  - 每条只做一个清晰子目标
- 你 = `人类多窗口执行面`
  - 负责开支线窗口
  - 在必要时帮我点击界面
  - 把支线截图或结果回传给主线

此外，一旦出现下面任意 `1` 条，我应停止硬磨，转为明确向你求助：

- 同一页面连续 `2` 次定位失败
- 预计还要继续烧 `8-10` 分钟以上
- 需要你点一下就能把信息压缩 `10` 倍以上
- 当前动作已经从分析退化成盲试 UI

以后复杂任务里，你不需要再临时发明协作方式；默认就走这套，而且我会更早止损。

## 4.2 固定窗口角色：总控台 / 运营主线 / 支线

从 `2026-03-15` 开始，长期协作里这三个窗口角色边界固定如下：

- `总控台`
  - 固定为当前窗口
  - 只负责裁决、统筹、路线、验真和收口
  - 不重新下沉到日常实现位
- `运营主线`
  - 固定为单独新开窗口
  - 只负责当天主线任务承接、推进、状态整理、支线消费
  - 不抢总控台的路线裁决权
- `支线`
  - 继续保持单目标合同
  - 只做一个清晰子目标
  - 不改主线状态，不擅自扩 scope

一句话记忆：

`总控台决定今天怎么打，运营主线负责今天怎么推，支线只负责各自那一块怎么做。`

只要这三个角色边界被混写，就会重新退化成“总控台既裁决又实现”的高摩擦模式。

## 4.3 每天 09:00 合并晨会

从 `2026-03-15` 开始，每天 `09:00` 固定增加一轮 `合并晨会`。

它不再只是 skill review，而是当天唯一正式的早间治理入口，同时完成：

- `总控台定盘`
- `运营主线起盘`
- `支线编排建议`
- `人机协作优化提案`

合并晨会的正式输出固定为这 `5` 段：

1. `总控台今日定盘`
2. `运营主线今日合同`
3. `支线编排建议`
4. `人机协作优化提案`
5. `评审吸收结果与唯一待裁决项`

各段口径固定如下：

- `总控台今日定盘`
  - 今日唯一主目标
  - 优先级顺序
  - stop condition
  - 明确不做什么
- `运营主线今日合同`
  - 今日由 `运营主线` 承接的主合同
  - 成功标准
  - 对支线的消费口径
- `支线编排建议`
  - 建议开的支线
  - 每条支线的单目标
  - 哪些支线不值得开
- `人机协作优化提案`
  - 哪些动作继续 AI 自治
  - 哪些动作应该更早切给人类
  - 哪些动作属于窗口协作优化，不属于体系重构
- `评审吸收结果与唯一待裁决项`
  - 这轮吸收了哪些 `daily review` 与 `governance review` 结论
  - 如果仍需裁决，只保留一个最小问题

## 4.4 它和 daily review / governance review 的关系

`daily review` 和 `governance review` 继续存在，但不再作为与晨会并列的第二入口。

固定关系如下：

- `daily review`
  - 负责系统盘点、结构变化、候选动作
  - 是 `合并晨会` 的输入，不是最终前台输出
- `governance review`
  - 负责真实性、成熟度、治理风险、失真诊断
  - 是 `合并晨会` 的输入，不是最终前台输出
- `09:00 合并晨会`
  - 负责把两类评审吸收到一个当天可执行的统一口径里
  - 只保留一个正式晨会入口，避免人类同时消费两套早间话语

因此，现有 automation 的正确升级方向不是再加一个平行晨会 automation，而是把已有 `09:00` review automation 升级为 `合并晨会` 入口。

## 5. 人类介入边界：必须打断你的场景 vs 不应打断你的场景

`AI大管家` 的默认原则是少打扰你，但不是无条件不打扰。

### 必须打断你的场景

- 登录
- 授权
- 付款
- 不可逆发布
- 不可逆删除
- 权限受阻且无法替代
- 不可替代的主观选择

### 原则上不应打断你的场景

- 工具选择
- 技能组合选择
- 一般执行分解
- 中间格式组织
- 常规证据整理
- 可验证的小决策

### 你的边界表达方式

高质量边界表达应像这样：

- “除登录、发布、删除、付款外，尽量少打扰我”
- “文风你先给默认版，只有价值判断冲突时再问我”
- “能用现有 skill 和脚本解决的，不要先让我选工具”

低质量边界表达通常像这样：

- “你先别动，什么都问我”
- “你自己看着办”
- “尽量做好”

前者边界清晰，后者会让系统要么过度打扰，要么假自治。

## 6. 高价值提法：任务输入模板、目标/边界/验真/闭环的表达方式

高价值输入的原则很简单：

- 少给局部动作，多给目标和约束
- 少给模糊评价，多给成功标准
- 少给工具偏好，多给验真要求
- 少把一次任务说成“一件事”，多把它说成“一轮回合”

### 标准任务提法

建议至少包含四项：

- `目标`
- `成功标准`
- `边界`
- `闭环要求`

示例：

```text
目标：帮我把 AI大管家的协作体系整理成长期主文档
成功标准：我以后提任务时能直接按文档协作，不需要再解释体系
边界：除结构路线冲突外，不必频繁打断我
闭环要求：做完后说明 gain / waste / next iterate
```

### 验真表达

不要只说“做好一点”“完整一点”，而要说：

- 结果怎么判断算完成
- 证据是什么
- 还有哪些开放问题是允许留下的

示例：

```text
验真要求：不是只给概念，要能落到我日常如何提任务、何时介入、何时放权
```

### 闭环表达

好的闭环要求会逼系统避免伪完成。

示例：

```text
闭环要求：最后给我一个可长期复用的入口文档，并说明最容易误用它的三个地方
```

## 7. 高频协作场景：建 skill、学新领域、知识库优先、系统评审、外部同步

这一章只讲你最常见的协作入口，不覆盖全部技能宇宙。

### 场景 A：建新 skill 或改 skill

你应该这样提：

```text
帮我创建/更新一个 skill，目标是 X，尽量复用现有模式，少打扰我。
```

`AI大管家` 应优先路由到：

- `skill-creator`

你主要看三件事：

- 这个 skill 是否真的必要
- 是否复用了现有结构
- 是否有清晰的验证口径

### 场景 B：学习一个陌生 API / 平台 / 方法

你应该这样提：

```text
帮我学一个陌生 API，先读官方说明书和攻略，再决定怎么做。
```

`AI大管家` 应优先路由到：

- `guide-benchmark-learning`

如果是 OpenAI 相关，再补：

- `openai-docs`

你要关注的不是“学了多少”，而是：

- 来源是否可靠
- 有没有 source-of-truth 与参考资料区分
- 学完后是否进入可执行状态

### 场景 C：先问知识库，再规划

你应该这样提：

```text
先问我的知识库，再帮我规划下一步。
```

`AI大管家` 应优先路由到：

- `knowledge-orchestrator`

你要关注：

- 原始问答是否先落地保存
- 后续规划是否真的建立在知识库回答上

### 场景 D：系统级能力盘点、治理复盘

你应该这样提：

```text
帮我做一次 AI大管家系统评审，看看技能体系和治理成熟度。
```

`AI大管家` 应优先调用：

- `review-skills`
- `review-governance`

这是系统级任务，不应误当成单一执行任务。

### 场景 E：外部同步

你应该这样理解：

- Feishu 是工作镜像
- GitHub 是协作与归档镜像
- 本地 artifacts 才是 canonical log

你要检查的不是“有没有同步”，而是：

- 本地记录是否已完整
- 是否先 dry-run 再 apply
- 镜像内容是否只反映最终真实状态

## 8. 反模式清单：把它当万能执行器、把镜像当真源、把动作完成当结果闭环

以下是三类高频误用。

### 反模式 1：把 AI大管家当万能执行器

症状：

- 你只丢动作，不给目标和边界
- 你要求它什么都亲自做
- 你忽略它的路由与治理价值

后果：

- 技能复用下降
- 协作质量退化
- 它从治理器退化成执行员

### 反模式 2：把 Feishu / GitHub 当真源

症状：

- 只看镜像，不看本地 canonical artifacts
- 以外部记录替代真实闭环判断

后果：

- 闭环状态失真
- 真实开放问题被覆盖
- 后续进化材料失真

### 反模式 3：把动作完成当结果闭环

症状：

- 命令跑完就当结束
- 文档写完就当完成
- 没有验证陈述与复盘

后果：

- 伪完成
- 伪理解
- 进化中断

### 反模式 4：过度微操

症状：

- 你替它指定每个工具和每一步
- 把所有小决策都拿回自己做

后果：

- 打断增多
- 自治下降
- 你自己成为系统瓶颈

### 附录 F：窗口 Heartbeat 最小写法

当任务已经进入 `总控台 / 运营主线 / 支线 / 卫星机支线` 并行协作时，每个窗口都可以落一条最小 heartbeat。

固定字段只有这 `8` 个：

- `window_id`
- `role`
- `current_phase`
- `last_action`
- `current_status`
- `latest_evidence_paths`
- `sole_blocker`
- `updated_at`

推荐写法如下：

```bash
python3 work/ai-da-guan-jia/scripts/ai_da_guan_jia.py window-heartbeat write \
  --window-id ops-mainline \
  --role 运营主线 \
  --current-phase "主线推进" \
  --last-action "更新 run 包并整理支线输入" \
  --current-status running \
  --evidence work/ai-da-guan-jia/artifacts/ai-da-guan-jia/runs/2026-03-15/<run-id>/summary.md \
  --sole-blocker none
```

总控台巡检固定用：

```bash
python3 work/ai-da-guan-jia/scripts/ai_da_guan_jia.py window-heartbeat inspect
python3 work/ai-da-guan-jia/scripts/ai_da_guan_jia.py window-heartbeat govern
```

读法固定为：

- `progressing`
  - 说明窗口还在推进
- `waiting_external`
  - 说明窗口明确在等外部条件
- `stuck`
  - 说明窗口已卡住，或 heartbeat 长时间没刷新
- `suspected_silent_lost`
  - 说明窗口可能静默失联，需要总控台介入

总控台的日内默认裁决顺序固定为：

1. 先跑 `window-heartbeat inspect`
2. 再看可信 evidence path
3. 再用 `window-heartbeat govern` 形成：
   - `继续`
   - `等待`
   - `裁决`
   - `收口`
4. 只有证据不足时，才请人类补充

自动发号施令只允许发生在：

- `继续`
  - 窗口仍在推进，且证据可信
- `等待 -> 继续`
  - 外部条件已消失，且不涉及人类边界
- `裁决`
  - blocker 属于总控可裁决的技术/范围问题
- `收口`
  - 窗口疑似静默失联，需要先回报状态

总控台不得自动越过：

- 登录、授权、支付
- 不可逆发布/删除
- 主观设计或业务取舍
- 多窗口总体优先级重排
- 证据不足时的完成升级
- 自动新开窗口
