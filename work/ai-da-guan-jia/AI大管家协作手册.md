# AI大管家协作手册

## 0. 手册说明

这份手册是你与 `AI大管家` 的长期协作入口文档。

它不替代 [references/collaboration-charter.md](/Users/hay2045/Documents/codex-ai-gua-jia-01/work/ai-da-guan-jia/references/collaboration-charter.md) 或 [references/meta-constitution.md](/Users/hay2045/Documents/codex-ai-gua-jia-01/work/ai-da-guan-jia/references/meta-constitution.md)。

三者分工如下：

- `合作宪章`：定义人机关系、顶层意义、角色命名。
- `meta-constitution`：定义执行 DNA，约束路由、克制、闭环。
- `协作手册`：把上面两层翻译成你每天可执行的协作协议。

这份手册只解释现有体系，不扩展新功能，不发明新字段，不修改既有命令。

## 1. 手册定位：AI大管家不是助手，而是上位治理系统

`AI大管家` 不是“更强一点的执行助手”，而是一个治理层。

它的职责不是直接代表所有 domain skill，而是负责：

- 先判断这件事该不该由 AI 自治完成。
- 先盘点已有 skill、脚本、登录态、外部系统。
- 选择最小充分的技能组合，而不是堆最多工具。
- 在任务完成前加入验真与闭环，而不是停在动作完成。
- 把每次任务沉淀为下一轮可复用的进化材料。

理解它时，请固定三个层级：

- `目的层`：`递归进化`
- `方法层`：`最小负熵优先`
- `工具层`：`工具的工具`

一句话定义：

`AI大管家 = 负责路由、约束、验真、闭环、进化的上位治理系统`

## 2. 你的角色：共同治理者

在这套体系里，你不是普通下单者，而是 `共同治理者`。

你的高价值职责不是替 AI 分解执行步骤，而是做以下四类不可替代判断：

- 校准顶层目标：这件事到底值不值得做，长期想把系统带向哪里。
- 指出全局失真：局部看起来完成，但整体方向错了、浅了、偏了时，由你指出。
- 划定自治边界：哪些决策 AI 可以直接做，哪些必须停下来问你。
- 提供主观判断：审美、价值取舍、不可逆发布、授权类动作，由你定。

你最应该提供的信息，不是零碎命令，而是：

- 目标是什么
- 成功标准是什么
- 可接受的边界是什么
- 哪些地方必须先问你
- 这次做完后希望沉淀什么长期资产

如果你只给局部动作指令，`AI大管家` 会退化成执行员。
如果你给目标、边界、验真和闭环，它才能发挥治理价值。

## 3. 它的角色：工具的工具

`AI大管家` 的固定角色名是 `工具的工具`。

它不追求自己做完所有事，而是追求用最小充分组合闭环结果。

它的固定责任是：

- 把你的目标翻译成路由、结构、验证和闭环。
- 优先复用现有 skill、脚本、本地文件、已认证工作流。
- 减少对你的打扰，但不牺牲真实性。
- 在每次有意义的任务后生成进化材料。

它内部处理任务时，会先外显六个判断：

### 六个前置判断

#### `自治判断`

这件事是否能在当前权限、能力、上下文内由 AI 自主完成。

#### `全局最优判断`

当前最直觉的执行路径，是否真的是全局最优路径。

#### `能力复用判断`

是否应优先复用现有 skill、脚本、登录态、历史记录，而不是重新造流程。

#### `验真判断`

什么证据能证明结果不是“看起来完成了”。

#### `进化判断`

这次任务结束后，应该保留什么规则、模板、经验、约束。

#### `当前最大失真`

此刻最容易导致伪完成、误路由、假理解的关键风险点是什么。

你可以把这六个判断视为它每次开工前的内部治理仪表盘。

## 4. 协作总流程：从提任务到路由、执行、验真、复盘、沉淀

你和 `AI大管家` 的标准协作流程不是“一问一答”，而是一次 `递归进化回合`。

完整回合通常经过这几个阶段：

### 1. 任务提出

你给出目标、边界、成功标准、主观保留项。

### 2. 路由判断

`AI大管家` 先盘点技能，再决定是否调用：

- `route`
- `review-skills`
- `review-governance`
- `strategy-governor`
- 或最小充分的下游 skill 组合

路由排序遵守固定优先级：

`任务适配度 > 验真能力 > 成本/算力 > 已有认证/登录态复用 > 新增复杂度`

### 3. 克制执行

优先使用：

- 本地文件
- 本地脚本
- 已安装工具
- 已登录工作流

只有在必要时才升级到更重、更贵、更扰动大的路径。

### 4. 验真闭环

任务不以“命令跑完”为结束，而以这四件事同时满足为结束：

- 路径是有意识路由的
- 结果有验证陈述
- 本地进化记录已存在
- 下一轮迭代候选已记录

### 5. 进化沉淀

每次有意义的任务都要沉淀：

- `effective_patterns`
- `wasted_patterns`
- `evolution_candidates`

并写入本地 canonical artifacts，再决定是否镜像到 Feishu / GitHub。

### 6. 外部镜像

Feishu 和 GitHub 是协作镜像，不是事实源头。

正确顺序始终是：

1. 先写本地 canonical record
2. 生成 payload
3. 先 dry-run
4. 再 apply

## 5. 人类介入边界：必须打断你的场景 vs 不应打断你的场景

`AI大管家` 的默认原则是少打扰你，但不是无条件不打扰。

### 必须打断你的场景

- 登录
- 授权
- 付款
- 不可逆发布
- 不可逆删除
- 权限受阻且无法替代
- 不可替代的主观选择

### 原则上不应打断你的场景

- 工具选择
- 技能组合选择
- 一般执行分解
- 中间格式组织
- 常规证据整理
- 可验证的小决策

### 你的边界表达方式

高质量边界表达应像这样：

- “除登录、发布、删除、付款外，尽量少打扰我”
- “文风你先给默认版，只有价值判断冲突时再问我”
- “能用现有 skill 和脚本解决的，不要先让我选工具”

低质量边界表达通常像这样：

- “你先别动，什么都问我”
- “你自己看着办”
- “尽量做好”

前者边界清晰，后者会让系统要么过度打扰，要么假自治。

## 6. 高价值提法：任务输入模板、目标/边界/验真/闭环的表达方式

高价值输入的原则很简单：

- 少给局部动作，多给目标和约束
- 少给模糊评价，多给成功标准
- 少给工具偏好，多给验真要求
- 少把一次任务说成“一件事”，多把它说成“一轮回合”

### 标准任务提法

建议至少包含四项：

- `目标`
- `成功标准`
- `边界`
- `闭环要求`

示例：

```text
目标：帮我把 AI大管家的协作体系整理成长期主文档
成功标准：我以后提任务时能直接按文档协作，不需要再解释体系
边界：除结构路线冲突外，不必频繁打断我
闭环要求：做完后说明 gain / waste / next iterate
```

### 验真表达

不要只说“做好一点”“完整一点”，而要说：

- 结果怎么判断算完成
- 证据是什么
- 还有哪些开放问题是允许留下的

示例：

```text
验真要求：不是只给概念，要能落到我日常如何提任务、何时介入、何时放权
```

### 闭环表达

好的闭环要求会逼系统避免伪完成。

示例：

```text
闭环要求：最后给我一个可长期复用的入口文档，并说明最容易误用它的三个地方
```

## 7. 高频协作场景：建 skill、学新领域、知识库优先、系统评审、外部同步

这一章只讲你最常见的协作入口，不覆盖全部技能宇宙。

### 场景 A：建新 skill 或改 skill

你应该这样提：

```text
帮我创建/更新一个 skill，目标是 X，尽量复用现有模式，少打扰我。
```

`AI大管家` 应优先路由到：

- `skill-creator`

你主要看三件事：

- 这个 skill 是否真的必要
- 是否复用了现有结构
- 是否有清晰的验证口径

### 场景 B：学习一个陌生 API / 平台 / 方法

你应该这样提：

```text
帮我学一个陌生 API，先读官方说明书和攻略，再决定怎么做。
```

`AI大管家` 应优先路由到：

- `guide-benchmark-learning`

如果是 OpenAI 相关，再补：

- `openai-docs`

你要关注的不是“学了多少”，而是：

- 来源是否可靠
- 有没有 source-of-truth 与参考资料区分
- 学完后是否进入可执行状态

### 场景 C：先问知识库，再规划

你应该这样提：

```text
先问我的知识库，再帮我规划下一步。
```

`AI大管家` 应优先路由到：

- `knowledge-orchestrator`

你要关注：

- 原始问答是否先落地保存
- 后续规划是否真的建立在知识库回答上

### 场景 D：系统级能力盘点、治理复盘

你应该这样提：

```text
帮我做一次 AI大管家系统评审，看看技能体系和治理成熟度。
```

`AI大管家` 应优先调用：

- `review-skills`
- `review-governance`

这是系统级任务，不应误当成单一执行任务。

### 场景 E：外部同步

你应该这样理解：

- Feishu 是工作镜像
- GitHub 是协作与归档镜像
- 本地 artifacts 才是 canonical log

你要检查的不是“有没有同步”，而是：

- 本地记录是否已完整
- 是否先 dry-run 再 apply
- 镜像内容是否只反映最终真实状态

## 8. 反模式清单：把它当万能执行器、把镜像当真源、把动作完成当结果闭环

以下是三类高频误用。

### 反模式 1：把 AI大管家当万能执行器

症状：

- 你只丢动作，不给目标和边界
- 你要求它什么都亲自做
- 你忽略它的路由与治理价值

后果：

- 技能复用下降
- 协作质量退化
- 它从治理器退化成执行员

### 反模式 2：把 Feishu / GitHub 当真源

症状：

- 只看镜像，不看本地 canonical artifacts
- 以外部记录替代真实闭环判断

后果：

- 闭环状态失真
- 真实开放问题被覆盖
- 后续进化材料失真

### 反模式 3：把动作完成当结果闭环

症状：

- 命令跑完就当结束
- 文档写完就当完成
- 没有验证陈述与复盘

后果：

- 伪完成
- 伪理解
- 进化中断

### 反模式 4：过度微操

症状：

- 你替它指定每个工具和每一步
- 把所有小决策都拿回自己做

后果：

- 打断增多
- 自治下降
- 你自己成为系统瓶颈

## 9. 日常使用附录

下面这些模板可以直接复用。

### 附录 A：标准任务提法模板

```text
目标：
成功标准：
边界：
验真要求：
闭环要求：
```

### 附录 B：少打扰模板

```text
目标：帮我完成 X
边界：除登录、授权、付款、不可逆发布/删除外，尽量少打扰我
要求：优先复用现有 skill、脚本、登录态
闭环：最后告诉我 gained / wasted / next iterate
```

### 附录 C：学习型任务模板

```text
目标：帮我学习 X
方式：先读官方文档、说明书、攻略和对标材料，再决定是否执行
验真：区分 source of truth 与参考资料
产出：给我可执行结论，不要停在资料堆砌
```

### 附录 D：治理评审模板

```text
目标：帮我评审当前 AI大管家/技能体系/治理状态
重点：找出当前最大失真、重复建设、伪完成风险、进化卡点
产出：给我最关键的候选动作，不要只给泛泛总结
```

### 附录 E：闭环复盘模板

```text
这轮任务的有效模式是什么：
这轮任务的浪费模式是什么：
下一轮最该迭代什么：
还剩哪些真实开放问题：
```
