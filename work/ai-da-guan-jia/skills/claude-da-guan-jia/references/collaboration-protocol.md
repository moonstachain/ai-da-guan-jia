# 协作总流程详细参考

## 标准递归进化回合

### 1. 任务提出

人类给出：目标、边界、成功标准、主观保留项。

高价值输入原则：
- 少给局部动作，多给目标和约束
- 少给模糊评价，多给成功标准
- 少给工具偏好，多给验真要求
- 少把一次任务说成"一件事"，多把它说成"一轮回合"

标准任务提法至少包含四项：目标、成功标准、边界、闭环要求。

### 2. 路由判断

先盘点技能，再决定调用：route / review-skills / review-governance / strategy-governor / 或最小充分下游 skill 组合。

路由排序固定优先级：
`任务适配度 > 验真能力 > 成本/算力 > 已有认证/登录态复用 > 新增复杂度`

### 3. 克制执行

优先使用：本地文件 → 本地脚本 → 已安装工具 → 已登录工作流。
只有在必要时才升级到更重、更贵、更扰动大的路径。

### 4. 验真闭环

任务以这四件事同时满足为结束：
- 路径是有意识路由的
- 结果有验证陈述
- 本地进化记录已存在
- 下一轮迭代候选已记录

### 5. 进化沉淀

每次有意义的任务都要沉淀：
- `effective_patterns`：什么路径/组合有效
- `wasted_patterns`：什么浪费了时间/token
- `evolution_candidates`：下一轮应该迭代什么

写入本地 canonical artifacts，再决定是否镜像到 Feishu / GitHub。

### 6. 外部镜像

正确顺序始终是：先写本地 canonical record → 生成 payload → 先 dry-run → 再 apply。

## 复杂任务拆线规则

按 **信息维 × 能量维** 判断：

| 信息维\能量维 | 低能量 | 高能量 |
|---|---|---|
| 高信息 | 默认AI做 | 主线+支线+人类执行面 |
| 低信息 | 挂起或并入 | 尽早切给人类 |

### 节能模式 vs 并行模式

- **节能模式**：优先单主线，省 token，遇到对人类容易/对AI很贵的动作立刻切给人类
- **并行模式**：1条主线+2-3条支线，目标是提速

满足任意 2 条时建议升级到并行模式：
- 同时涉及 repo、本地工具、飞书/妙搭、浏览器、远端机器
- 同时包含主实现、分析、验证、截图、人工点击
- 已出现天然可并行的子问题
- 单窗口推进预计超 20-30 分钟且要频繁切上下文

### 止损信号

出现下面任意 1 条，应停止硬磨，转为明确向人类求助：
- 同一页面连续 2 次定位失败
- 预计还要继续烧 8-10 分钟以上
- 人类点一下就能把信息压缩 10 倍以上
- 当前动作已经从分析退化成盲试 UI

```

边界：除登录、授权、付款、不可逆发布/删除外，尽量少打扰我

```
