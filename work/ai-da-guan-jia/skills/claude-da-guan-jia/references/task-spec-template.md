# Task Spec 标准模板与编号规则

## 编号规则

### 命名空间隔离

| 类型 | 编号格式 | 示例 |
|------|----------|------|
| 常规任务 | `TS-{模块缩写}-{序号}` | TS-KB-07, TS-DASH-04 |
| 项目级任务 | `PROJ-{项目名}-{序号}` | PROJ-V2-CLONE-01 |
| 客户项目任务 | `TS-{客户缩写}-{序号}` | TS-BRM-01 |

### 已占用编号（不可复用）

- `TS-V2-01`：已占用（INIT模板化，已完成）
- `TS-V2-*` 系列：V2-CLONE 项目不可使用，必须用 `PROJ-V2-CLONE-*`

---

## 完整模板

```markdown
# Task Spec: [TASK-ID] [标题]

> 状态：Draft | Ready | In-Progress | Blocked | Done
> 负责人：Codex
> 审批人：人类
> 创建日期：YYYY-MM-DD
> 关联 INIT 版本：V[XX]

## 目标

一句话说清楚要达成什么。不超过两行。

## 背景

为什么现在要做这件事。与当前阶段目标的关系。
引用 INIT 中的相关段落（如有）。

## 范围

### 包含
- 明确列出要做的事

### 不包含
- 明确列出不做的事（防范围蔓延）

## 输入

人类/Claude 提供给 Codex 的所有素材：
- 文件路径
- API 凭证（如需）
- 参考文档
- 上下文信息

## 输出物

Codex 完成后必须交付的文件/数据清单：

| 输出物 | 格式 | 存放位置 |
|--------|------|----------|
| ... | ... | ... |

## 验收标准

每条标准必须可独立验证：

- [ ] 标准1（验真方式：xxx）
- [ ] 标准2（验真方式：xxx）
- [ ] ...

## 执行步骤

按顺序列出 Codex 需要执行的步骤：

1. Step 1：...
2. Step 2：...
3. ...

## 约束与风险

### 约束
- 不能做什么

### 风险
- 可能出什么问题
- 降级方案

## 最后一步（强制 — 不可遗漏）

1. 是否更新 `CLAUDE-INIT.md` → [是/否，说明更新哪些段落]
2. 是否更新飞书总控或其他 mirror → [是/否，说明哪个表]
3. 失败时是否输出兜底文件 → [是/否，兜底文件内容说明]
4. 最终回传什么 evidence 给 Claude → [具体说明]
5. 【强制】同步飞书战略任务追踪表 → `tblB9JQ4cROTBUnr`
   - 字段：Task ID / 标题 / 状态 / 产出物路径 / 完成日期
```

## Task Spec 质量检查清单

输出 Task Spec 前，逐条自检：

- [ ] 目标是否一句话说清？
- [ ] 范围的"不包含"是否明确？
- [ ] 每个验收标准是否附带验真方式？
- [ ] 执行步骤是否足够具体，Codex 不需要猜？
- [ ] 最后一步五项是否全部填写？
- [ ] 编号是否与已有编号不冲突？
- [ ] 没有把 planning-only 内容混入执行步骤？
