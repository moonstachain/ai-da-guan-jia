# Agent 三代演进定位

更新时间：2026-03-17  
定位：解释原力OS为什么把自己判断为“第二代末期 → 第三代入口”。

## 总体框架

这里的“三代”不是 marketing 标签，而是治理单元成熟度的分层。

### 第一代：手工拼接期

核心特点：

- 单轮对话驱动
- 强依赖人工串行传递
- 结构靠提示词和人工记忆维持

在原力OS中的对应区间：

- `R0-R5`

直观表现：

- Claude 出 spec
- Codex 单点执行
- 体系还没有稳定的多层记忆、镜像合同和治理骨架

## 第二代：模块组装期

核心特点：

- 多仓库
- 多模块
- 多表
- 多镜像面
- 局部模板化

在原力OS中的对应区间：

- `R6-R17`

直观表现：

- Skill 体系成形
- 飞书、多维表、妙搭、GitHub 同时进入系统
- `AI大管家` 开始形成治理内核
- 14 件套工件、治理审计、编年史、R18 规划等逐步出现

第二代的价值在于：

- 系统不再只靠单轮对话
- 开始具备结构化记忆、执行链路和可审计性

第二代的问题也很明显：

- 共享层、实例层、镜像层经常混写
- `INIT` 很容易角色过载
- 很多结构仍然依赖“理解这套架构的人”才能运转

## 第三代：Agent 自治期

核心特点：

- 治理单元可以实例化
- 用户更像在“用产品”，而不是“理解底层后操作系统”
- 同事/客户可以拿到结构化模板而不是口头传承

在原力OS中的目标实例：

- `V1 多机指数化`
- `V2 同事复制`
- `V3 客户同构`

第三代不等于“全自动无人工”。

第三代真正的判断标准是：

- 角色分工清楚
- 启动恢复简单
- 模板能用
- 新实例能起来
- 人类只在目标、边界和不可逆动作上介入

## 当前定位

当前定位不是“已经第三代”，而是：

**第二代末期 → 第三代入口**

这么判断的原因是：

- 第二代骨架已经形成
- 多层记忆模型已经开始被合同化
- source-of-truth 已经开始对象级固定
- `memory.md` 和 `INIT` 四层分离已进入正式建设

但还没有完全进入第三代，因为：

- 共享层模板还没彻底定型
- GitHub 分发基座还没完成规范化
- 同事/客户还不能在不理解底层架构的前提下直接拿到实例

## 跃迁标志

从第二代进入第三代，至少要看到这些信号：

1. 新实例能按模板初始化，而不是靠人工口述
2. `CLAUDE-INIT`、`memory.md`、合同层、执行镜像层边界稳定
3. GitHub / Feishu 不再被误当成 runtime truth
4. 同事拿到实例后只需要操作产品，而不是学习全部底层对象模型
5. 客户实例和内部实例共享一套可治理的模板骨架

## 借鉴来源

这一判断吸收了知识图谱 PPT Agent 产品的演进研究，尤其是：

- 从手工 prompt 到流程编排
- 从串行单点到并行 pipeline
- 从模型调用到可治理的 agent orchestration

但原力OS没有直接照搬，而是把它翻译成自己的治理语境：

- Claude = 策略脑
- AI大管家 = 知行脑
- Codex = 主要执行器
- 人类 = 共同治理者和最终裁决者

## 这份定位的实际用途

这份文档不是抽象说明书，而是帮助后续每轮判断：

- 当前是在补第二代骨架，还是已经在做第三代模板化
- 某项工作该进入合同层、模板层，还是仍停留在实例层
- 哪些东西应该继续制度化，哪些只是局部实验
