# 三向量扩展架构

更新时间：2026-03-17  
定位：这是 `R18` 阶段的战略长文，承接 `CLAUDE-INIT.md` 中不再适合保留的三向量详细论述。

## 统一认知

原力OS当前把一个最小可扩展治理单元定义为：

`Claude + Codex + 飞书协作面 + GitHub 分发面 + 人类裁决`

这不是五个松散工具，而是一个 `Governance Unit`。  
三向量扩展不是三个独立项目，而是同一个治理单元的三种实例化方式：

- `V1 多机指数化`
- `V2 同事复制`
- `V3 客户同构`

核心工程只有一个：

**把治理单元做成可实例化、可恢复、可审计、可分发的模板。**

## 基础原则

### 1. 向内更真，向外更协作

- 本地 artifacts 是 runtime truth 的 canonical
- `AI大管家` 合同与 `memory.md` 是 machine-side institutional memory
- GitHub 是 shared startup memory 和执行镜像分发层
- Feishu 是人类前台和 cockpit 层

### 2. 共享层与实例层必须分离

共享层负责：

- shared skills
- 审计标准
- 模板
- 启动记忆分发

实例层负责：

- 各自的飞书 Base
- 各自的驾驶舱
- 各自的 Codex / 环境
- 各自的业务数据

### 3. GitHub 是分发基座，不只是归档

扩展速度取决于共享层是否干净、可 clone、可解释。

如果 GitHub 本身是脏的：

- 多机会放大混乱
- 同事复制会继承失真
- 客户同构会把局部问题产品化

## V1：多机指数化

### 目标

让同一个人同时操控多个治理单元，把算力和执行窗口指数化。

### 核心形态

- 一个 Claude 策略脑
- 多个 Codex 执行节点
- 主机 / 卫星机 / 虚机并行
- 人类做批量验收，而不是串行传话

### 关键任务

- 卫星机治理对齐
- 机器注册表与心跳
- 任务分发路由
- 标准化部署脚本

### 关键风险

- 机器身份和状态不统一
- 主线 / 支线 / 卫星的 canonical 边界混乱
- 批量并行下镜像面先于 canonical 更新

## V2：同事复制

### 目标

让不同同事各自拥有一个独立治理单元，但共享同一套治理语言、模板和审计标准。

### 核心形态

- 每个同事有自己的 Base / dashboard / INIT / 环境
- 共享 skills-pack、治理合同、审计标准
- 同事不需要理解底层架构，只需要在模板化产品上工作

### 关键任务

- `CLAUDE-INIT` 模板化
- 飞书 Base 初始化脚本
- 妙搭模板化
- 模块拆分矩阵
- onboarding SOP

### 关键风险

- 同事直接改共享层
- 不同实例把镜像面当成 canonical
- 模板复制的是表面，不是治理骨架

## V3：客户同构

### 目标

把治理单元产品化为客户可用的行业定制实例。

### 核心形态

- 行业模板
- 客户初始化工具链
- 定制驾驶舱
- 服务分层交付

### 交付层级

- 基础版：初始化与模板交付
- 陪伴版：定期审计与治理陪跑
- 深度版：Codex 帮执行与闭环

### 关键风险

- 客户实例直接写回共享层
- 行业模板未固化就对外复制
- pricing / SOP / onboarding 不成体系

## 共享层 vs 实例层

| 层级 | 内容 | 权限 | 说明 |
|---|---|---|---|
| 共享层 | `skills-pack`、治理合同、审计标准、妙搭模板、`CLAUDE-INIT` 模板 | 只读共享 | 任何改动都应走治理流程 |
| 实例层 | Base、dashboard、环境、个人 INIT、业务数据 | 实例独立读写 | 归各节点所有者管理 |

## Pipeline 并行模型

三向量扩展默认不再把“串行传递”当基本模式。

目标状态是：

- Claude 可以继续设计后续 spec
- Codex-A 执行任务 A
- Codex-B 执行任务 B
- 人类集中验收关键结果

这要求：

- spec 必须足够 decision-complete
- human boundary 必须精确
- canonical / mirror 更新顺序必须清楚

## 多模型编排

多机扩展的自然下一步是多模型编排：

- 策略分析适合 Claude
- 代码执行适合 Codex / GPT-5.4
- 图像与某些中文内容可路由到其他模型

关键点不是“模型越多越好”，而是：

- 路由规则清楚
- evidence 回传格式统一
- canonical 归档仍然回到同一治理骨架

## GitHub 治理前置要求

三向量扩展前，GitHub 需要先完成治理盘点与规范化：

- 仓库命名和角色清晰
- branch / tag / issue / Project 规则统一
- dirty workspace 与历史包袱被识别和收束

否则三向量只会复制混乱。

## 进入第三代的判断标准

当满足以下条件时，系统才真正从“第二代末期”跨到“第三代入口”：

- 同事或客户拿到实例后，不需要理解底层架构
- 共享层和实例层稳定分离
- 启动记忆、制度记忆、canonical、镜像面边界清晰
- 新实例的初始化和治理基线是模板化的，而不是一次次手工口述
